# Ch.09 외부 연동 — Gateway와 웹훅

> ⚠️ **개발 경험 필요.** 비개발자는 기본 텔레그램/Slack 연동(Ch.04)만으로도 충분합니다.

> "에이전트가 아무리 똑똑해도, 바깥세상과 소통 못 하면 방 안의 천재일 뿐이다."

이번 장에서는 OpenClaw 에이전트를 외부 서비스(채널톡, Slack, 텔레그램 등)와 연결하는 방법을 다룹니다. Gateway의 구조부터 웹훅 설정, 실전 CS 자동답변 파이프라인까지 하나씩 짚어보겠습니다.

---

## 9.1 Gateway란 무엇인가

Gateway는 에이전트의 **현관문 + 경비원 + 우체통**입니다.

```
┌─────────────────────────────────────────────┐
│                  Gateway                     │
│          (단일 데몬, 포트 18789)              │
│                                              │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│   │ WebSocket│  │   HTTP   │  │  Webhook │  │
│   │  (채팅)  │  │  (API)   │  │ (수신)   │  │
│   └────┬─────┘  └────┬─────┘  └────┬─────┘  │
│        │             │             │         │
│        └─────────────┼─────────────┘         │
│                      ▼                       │
│              에이전트 라우팅                    │
└─────────────────────────────────────────────┘
```

핵심 포인트:

- **단일 데몬**: 하나의 프로세스가 모든 통신을 처리합니다
- **하나의 포트**: `18789` 하나로 WebSocket, HTTP, webhook, cron 전부 처리
- **라우팅**: 들어온 요청을 적절한 에이전트에게 전달

Gateway를 띄우는 건 간단합니다:

```bash
# Gateway 시작
openclaw gateway start

# 상태 확인
openclaw gateway status

# 로그 확인
openclaw gateway logs --tail 50
```

Gateway가 실행되면 `http://localhost:18789`에서 모든 요청을 받습니다.

## 9.2 웹훅(Hooks) 설정

외부 서비스가 에이전트에게 메시지를 보내려면 **웹훅 매핑**이 필요합니다. `openclaw.json`에 설정합니다:

```json
{
  "hooks": {
    "mappings": {
      "channeltalk-new-chat": {
        "agentId": "main",
        "sessionKey": "hook:ingress",
        "transform": "channeltalk-flatten.js"
      }
    }
  }
}
```

각 필드를 하나씩 살펴봅시다:

| 필드 | 설명 | 예시 |
|------|------|------|
| `channeltalk-new-chat` | 웹훅 엔드포인트 이름. URL 경로가 됩니다 | `POST /hooks/channeltalk-new-chat` |
| `agentId` | 이 웹훅을 받을 에이전트 ID | `"main"`, `"cs-bot"` |
| `sessionKey` | 세션 식별자. 같은 키면 같은 대화로 이어짐 | `"hook:ingress"` |
| `transform` | 외부 데이터를 변환할 JS 파일 | `"channeltalk-flatten.js"` |

여러 웹훅을 동시에 설정할 수도 있습니다:

```json
{
  "hooks": {
    "mappings": {
      "channeltalk-new-chat": {
        "agentId": "cs-agent",
        "sessionKey": "hook:channeltalk",
        "transform": "channeltalk-flatten.js"
      },
      "shopify-order": {
        "agentId": "order-agent",
        "sessionKey": "hook:shopify",
        "transform": "shopify-flatten.js"
      },
      "slack-mention": {
        "agentId": "main",
        "sessionKey": "hook:slack",
        "transform": "slack-flatten.js"
      }
    }
  }
}
```

웹훅이 호출되면 이런 흐름으로 동작합니다:

```
외부 서비스 → POST /hooks/{매핑이름} → transform 실행 → 에이전트에 메시지 전달
```

## 9.3 Transform 함수 — 데이터 번역기

외부 서비스가 보내는 JSON은 보통 복잡합니다. 채널톡 웹훅 페이로드만 해도 수십 개의 필드가 들어있죠. Transform 함수는 이 데이터를 **에이전트가 이해할 수 있는 형태로 변환**합니다.

### 기본 구조

```javascript
// transforms/channeltalk-flatten.js

/**
 * @param {object} payload - 외부 서비스가 보낸 원본 JSON
 * @returns {object|null} - 에이전트에 전달할 메시지 객체, null이면 무시
 */
module.exports = function transform(payload) {
  // 1. 봇/매니저 메시지는 무시 (무한루프 방지)
  if (payload.sender?.type === "bot" || payload.sender?.type === "manager") {
    return null; // null을 반환하면 에이전트에 전달하지 않음
  }

  // 2. 핵심 정보만 추출
  const customerName = payload.sender?.name || "고객";
  const message = payload.message?.plainText || "";
  const chatId = payload.chat?.id || "unknown";
  const channelId = payload.channel?.id || "";

  // 3. 읽기 좋은 메시지로 생성
  return {
    text: `[채널톡 문의] ${customerName}: ${message}`,
    metadata: {
      source: "channeltalk",
      chatId: chatId,
      channelId: channelId,
      customerName: customerName,
      originalTimestamp: payload.message?.createdAt
    }
  };
};
```

### 왜 봇/매니저 메시지를 필터링하나요?

에이전트가 채널톡에 답변을 보내면, 그 답변 자체가 다시 웹훅으로 돌아옵니다. 필터링하지 않으면:

```
고객 문의 → 에이전트 답변 → 웹훅으로 다시 수신 → 에이전트가 또 답변 → 무한루프!
```

이걸 막으려면 반드시 `sender.type`을 체크해야 합니다.

### 실전 예제: 채널톡 전체 transform

```javascript
// transforms/channeltalk-flatten.js (실전 버전)

module.exports = function transform(payload) {
  const event = payload.event || payload.refers?.event;

  // 지원하는 이벤트만 처리
  const ALLOWED_EVENTS = ["chat.created", "message.created"];
  if (!ALLOWED_EVENTS.includes(event)) {
    return null;
  }

  // 봇/매니저 메시지 필터링
  const senderType = payload.sender?.type
    || payload.refers?.message?.personType;
  if (senderType === "bot" || senderType === "manager") {
    return null;
  }

  // push 이벤트 무시 (뒤에서 자세히 설명)
  if (payload.refers?.message?.requestType === "push") {
    return null;
  }

  // 핵심 정보 추출
  const message = payload.refers?.message?.plainText
    || payload.message?.plainText
    || "";

  const customerName = payload.refers?.user?.name
    || payload.sender?.name
    || "고객";

  const phone = payload.refers?.user?.mobileNumber || "";
  const chatId = payload.refers?.chat?.id
    || payload.chat?.id
    || "unknown";

  // 빈 메시지 무시
  if (!message.trim()) {
    return null;
  }

  return {
    text: [
      `[채널톡 CS 문의]`,
      `고객명: ${customerName}`,
      `연락처: ${phone ? phone : "(비로그인)"}`,
      `채팅ID: ${chatId}`,
      `---`,
      `${message}`
    ].join("\n"),
    metadata: {
      source: "channeltalk",
      chatId,
      customerName,
      phone
    }
  };
};
```

### Transform 파일 위치

프로젝트 루트의 `transforms/` 폴더에 놓습니다:


## 9.4 실전: CS 자동답변 파이프라인

가장 흔한 사용 사례인 **채널톡 CS 자동답변**을 처음부터 끝까지 구축해봅시다.

### 전체 흐름

```
고객 문의 → 채널톡 → n8n(토큰 주입) → OpenClaw webhook
                                         ↓
                                      transform
                                         ↓
                                      에이전트
                                         ↓
                                   API로 채널톡 답변
                                         ↓
                                   Slack에 보고
```

### 왜 n8n이 중간에 필요한가요?

채널톡의 웹훅 설정에는 **Authorization 헤더를 추가하는 기능이 없습니다**. 그런데 OpenClaw Gateway는 인증 토큰이 없는 요청을 거부합니다.

해결책: **n8n을 프록시로 사용**하여 Bearer 토큰을 주입합니다.

```
채널톡 웹훅
  │
  │  (Authorization 헤더 없음)
  ▼
n8n Webhook 수신
  │
  │  (Authorization: Bearer {토큰} 추가)
  ▼
OpenClaw Gateway
```

### Step 1: n8n 웹훅 노드 설정

n8n에서 새 워크플로우를 만듭니다:

```
[Webhook 트리거] → [HTTP Request 노드]
```

**Webhook 트리거 설정:**
- Method: POST
- Path: `/channeltalk-webhook`

**HTTP Request 노드 설정:**
- Method: POST
- URL: `http://your-gateway:18789/hooks/channeltalk-new-chat`
- Headers:
  - `Authorization`: `Bearer YOUR_64_CHAR_HEX_TOKEN`
  - `Content-Type`: `application/json`
- Body: `{{ $json }}` (수신한 페이로드 그대로 전달)

n8n의 HTTP Request 노드 설정을 JSON으로 보면:

```json
{
  "nodes": [
    {
      "name": "Webhook",
      "type": "n8n-nodes-base.webhook",
      "parameters": {
        "httpMethod": "POST",
        "path": "channeltalk-webhook",
        "responseMode": "responseNode"
      }
    },
    {
      "name": "Forward to OpenClaw",
      "type": "n8n-nodes-base.httpRequest",
      "parameters": {
        "method": "POST",
        "url": "http://localhost:18789/hooks/channeltalk-new-chat",
        "sendHeaders": true,
        "headerParameters": {
          "parameters": [
            {
              "name": "Authorization",
              "value": "Bearer a1b2c3d4e5f6..."
            }
          ]
        },
        "sendBody": true,
        "bodyParameters": {
          "parameters": [
            {
              "name": "={{$json}}",
              "value": ""
            }
          ]
        }
      }
    }
  ]
}
```

### Step 2: 채널톡에서 n8n 웹훅 URL 등록

채널톡 설정 > 웹훅에서:

```
URL: https://your-n8n-domain.com/webhook/channeltalk-webhook
이벤트: chat.created, message.created
```

### Step 3: OpenClaw 에이전트 설정

CS 전용 에이전트를 만듭니다:

```markdown
<!-- agents/cs-agent/AGENTS.md -->

# CS 자동답변 에이전트

## 역할
채널톡으로 들어오는 고객 문의에 자동으로 답변합니다.

## 규칙
1. Q&A 데이터베이스를 반드시 먼저 검색할 것
2. 데이터베이스에 없는 질문은 "담당자에게 확인 후 안내드리겠습니다"로 응답
3. 개인정보(주문번호, 연락처 등)는 비로그인 고객에게 절대 알려주지 않을 것
4. 모든 답변은 Slack #cs-log 채널에 보고할 것
```

에이전트의 스킬에서 채널톡 API를 호출합니다:

```javascript
// skills/reply-channeltalk/index.js

async function replyToChat(chatId, message, channelId) {
  const response = await fetch(
    `https://api.channel.io/open/v5/channels/${channelId}/user-chats/${chatId}/messages`,
    {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "X-Access-Key": process.env.CHANNELTALK_ACCESS_KEY,
        "X-Access-Secret": process.env.CHANNELTALK_ACCESS_SECRET,
      },
      body: JSON.stringify({
        dto: {
          blocks: [
            {
              type: "text",
              value: message,
            },
          ],
        },
      }),
    }
  );

  return response.ok;
}
```

### Step 4: Slack 보고

에이전트가 답변을 보낸 후 Slack에 로그를 남깁니다:

```javascript
// skills/notify-slack/index.js

async function notifySlack(channel, text) {
  await fetch("https://slack.com/api/chat.postMessage", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${process.env.SLACK_BOT_TOKEN}`,
    },
    body: JSON.stringify({
      channel: channel,  // "#cs-log"
      text: text,
    }),
  });
}

// 사용 예시
await notifySlack("#cs-log", [
  `:speech_balloon: *CS 자동답변 완료*`,
  `고객: ${customerName}`,
  `문의: ${question}`,
  `답변: ${answer}`,
  `출처: ${source}`,  // Q&A DB or 에이전트 판단
].join("\n"));
```

## 9.5 Gateway 캐시 주의사항

Gateway는 성능을 위해 transform 파일을 **캐시**합니다. transform 코드를 수정했는데 변화가 없다면:

```bash
# Gateway 재시작 (캐시 초기화)
openclaw gateway restart

# 또는 개발 중에는 --no-cache 옵션 사용
openclaw gateway start --no-cache
```

> **삽질 사례 #1**: transform 로직을 바꿨는데 계속 예전 동작을 했다면, 십중팔구 캐시 문제입니다. 개발 중에는 반드시 `--no-cache`를 쓰거나, 수정할 때마다 재시작하세요.

## 9.6 Tailscale 연동 — 외부에서 안전하게 접속

로컬에서 돌아가는 Gateway를 외부에서 접속하려면 **Tailscale**이 가장 안전한 선택입니다.

### Serve vs Funnel 비교

| 구분 | Tailscale Serve | Tailscale Funnel |
|------|----------------|-----------------|
| 접근 범위 | Tailnet 내부만 | 인터넷 전체 |
| 인증 | Tailscale 로그인만으로 충분 | **반드시 비밀번호/토큰 필요** |
| 용도 | 팀원끼리 공유 | 외부 서비스 웹훅 수신 |
| 보안 수준 | 높음 | 주의 필요 |

### Serve 설정 (tailnet 내부용)

```bash
# Gateway를 tailnet에 노출
tailscale serve --bg https+insecure://localhost:18789

# 확인
tailscale serve status
# https://my-machine.tail12345.ts.net/ -> https://localhost:18789
```

이제 같은 tailnet에 있는 팀원은 `https://my-machine.tail12345.ts.net/`으로 접속할 수 있습니다.

### Funnel 설정 (인터넷 노출)

```bash
# 인터넷에 노출 (주의!)
tailscale funnel --bg https+insecure://localhost:18789

# 확인
tailscale funnel status
# https://my-machine.tail12345.ts.net/ -> https://localhost:18789
# (인터넷에서 접근 가능)
```

> **중요**: Funnel을 사용하면 누구나 접속할 수 있으므로, 반드시 Gateway에 **인증 토큰을 설정**하세요.

```json
{
  "gateway": {
    "auth": {
      "type": "bearer",
      "token": "YOUR_64_CHAR_HEX_TOKEN"
    }
  }
}
```

토큰 생성:

```bash
# 64자 hex 토큰 생성
openssl rand -hex 32
# 예: a1b2c3d4e5f678901234567890abcdef...
```

## 9.7 안전장치 3가지

외부 연동은 곧 **외부 입력을 받는다**는 의미입니다. 반드시 안전장치를 마련하세요.

### 1. 비로그인 고객 개인정보 차단

```markdown
<!-- agents/cs-agent/AGENTS.md 에 추가 -->

## 개인정보 보호 규칙
- 비로그인 고객(연락처가 없는 고객)에게는 절대로 다음 정보를 알려주지 마세요:
  - 주문번호, 배송 상태, 결제 내역
  - 다른 고객의 어떤 정보도
- 비로그인 고객이 개인정보를 요청하면:
  "본인 확인이 필요한 정보입니다. 로그인 후 다시 문의해주세요."
```

### 2. 확실하지 않으면 사람에게 넘기기

```markdown
## 에스컬레이션 규칙
다음 상황에서는 반드시 담당자에게 넘기세요:
- Q&A 데이터베이스에 없는 질문
- 환불/교환/클레임 관련
- 고객이 화난 상태 (감정 키워드 감지)
- 기술적 오류 신고

에스컬레이션 방법:
1. 고객에게: "담당자에게 확인 후 빠르게 안내드리겠습니다."
2. Slack #cs-escalation 채널에 상세 내용 전달
```

### 3. 기존 Q&A 전수 대조

에이전트가 답변하기 전에 반드시 Q&A 데이터베이스를 검색합니다:

```markdown
## 답변 프로세스
1. 고객 문의 수신
2. Q&A 데이터베이스 검색 (유사도 0.8 이상)
3-a. 매칭되면: 해당 답변 사용 (출처 명시)
3-b. 매칭 안 되면: 담당자에게 에스컬레이션
4. 절대로 Q&A에 없는 내용을 지어내지 않을 것
```

## 9.8 4가지 접속 경로

OpenClaw 에이전트에 접속하는 경로를 정리하면:

```
┌────────────────────────────────────────────────────┐
│                    에이전트                          │
│                                                    │
│  ① Slack ──────────────── 팀 내부 소통              │
│  ② 텔레그램 ──────────── 모바일 접속                 │
│  ③ WebChat(로컬) ─────── 개발/테스트                 │
│  ④ Tailscale+WebChat ── 외부 안전 접속               │
│                                                    │
│  + Webhook ────────────── 외부 서비스 연동           │
└────────────────────────────────────────────────────┘
```

| 경로 | 설정 방법 | 용도 |
|------|----------|------|

## 9.9 Gateway 데몬: macOS launchd

Mac에서 Gateway를 항상 실행 상태로 유지하려면 **launchd**를 사용합니다.

```xml
<!-- ~/Library/LaunchAgents/com.openclaw.gateway.plist -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.openclaw.gateway</string>

    <key>ProgramArguments</key>
    <array>
        <string>/usr/local/bin/openclaw</string>
        <string>gateway</string>
        <string>start</string>
        <string>--foreground</string>
    </array>

    <key>WorkingDirectory</key>
    <string>/Users/yourname/my-openclaw-project</string>

    <key>RunAtLoad</key>
    <true/>

    <key>KeepAlive</key>
    <true/>

    <key>StandardOutPath</key>
    <string>/tmp/openclaw-gateway.stdout.log</string>

    <key>StandardErrorPath</key>
    <string>/tmp/openclaw-gateway.stderr.log</string>
</dict>
</plist>
```

등록 및 관리:

```bash
# 등록
launchctl load ~/Library/LaunchAgents/com.openclaw.gateway.plist

# 상태 확인
launchctl list | grep openclaw

# 중지
launchctl unload ~/Library/LaunchAgents/com.openclaw.gateway.plist

# 로그 확인
tail -f /tmp/openclaw-gateway.stdout.log
```

이렇게 하면 Mac을 재시작해도 Gateway가 자동으로 올라옵니다. 크래시가 나도 `KeepAlive` 설정 덕분에 자동 재시작됩니다.

## 9.10 실전 삽질 사례 3가지

### 삽질 #1: 인증 헤더가 안 붙는다

**증상**: 채널톡에서 웹훅을 보냈는데 Gateway가 `401 Unauthorized` 반환

**원인**: 채널톡은 웹훅 설정에 커스텀 헤더(Authorization)를 추가하는 기능을 제공하지 않습니다.

**해결**: n8n을 프록시로 사용하여 Authorization 헤더를 주입합니다(9.4절 참고).

```
# 잘못된 접근
채널톡 → (헤더 없음) → OpenClaw Gateway → 401!

# 올바른 접근
채널톡 → n8n → (Bearer 토큰 주입) → OpenClaw Gateway → 200 OK
```

### 삽질 #2: push 이벤트로 무한루프

**증상**: 하나의 고객 문의에 에이전트가 2-3번 답변

**원인**: 채널톡은 `message.created` 이벤트를 보낼 때, `requestType: "push"`인 알림성 이벤트도 함께 보냅니다. 이걸 처리하면 같은 메시지를 중복 처리하게 됩니다.

**해결**: transform에서 push 이벤트를 필터링합니다.

```javascript
// transform 함수 안에서
if (payload.refers?.message?.requestType === "push") {
  return null;  // push 이벤트는 무시
}
```

### 삽질 #3: Tailscale Funnel 포트 불일치

**증상**: Tailscale Funnel 설정했는데 외부에서 접속 불가

**원인**: Funnel은 기본적으로 443 포트를 사용하는데, Gateway가 18789에서 돌고 있음. 포트 매핑이 빠짐.

**해결**: Funnel 설정 시 로컬 포트를 정확히 지정합니다.

```bash
# 잘못된 설정 (포트 미지정)
tailscale funnel --bg https://localhost

# 올바른 설정 (Gateway 포트 명시)
tailscale funnel --bg https+insecure://localhost:18789
```

확인 방법:

```bash
tailscale funnel status
# 다음과 같이 나와야 정상:
# https://my-machine.tail12345.ts.net/
#   |-- proxy https://localhost:18789
```

## 9.11 정리

이번 장에서 다룬 내용을 정리합니다:

```
Gateway = 현관문 + 경비원 + 우체통
  ├── 단일 데몬, 포트 18789
  ├── WebSocket / HTTP / Webhook / Cron 통합 처리
  └── transform으로 외부 데이터 변환

웹훅 설정 = openclaw.json의 hooks.mappings
  ├── agentId: 받을 에이전트
  ├── sessionKey: 세션 식별
  └── transform: 데이터 변환 스크립트

외부 노출 = Tailscale
  ├── Serve: tailnet 내부 (안전)
  └── Funnel: 인터넷 노출 (토큰 필수)

안전장치 3가지
  ├── 비로그인 고객 개인정보 차단
  ├── 불확실하면 사람에게
  └── Q&A 전수 대조
```

다음 장에서는 이렇게 외부에 노출된 에이전트를 **안전하게 지키는 보안** 방법을 다룹니다.

## 심화: 웹훅 보안 하드닝

프로덕션 환경에서 웹훅을 운영할 때는 기본 설정 이상의 보안 강화가 필요합니다. 다음은 하드닝된 hooks 설정입니다:

```json5
hooks: {
  enabled: true,
  token: "64-char-hex-minimum-length-token",
  path: "/hooks",
  mappings: [{
    match: { path: "channeltalk" },
    action: "agent",
    agentId: "external",       // 외부 전용 에이전트로 라우팅
    sessionKey: "hook:ingress"  // 고정 세션키
  }]
}
```

### 하드닝 규칙

- **hooks.token은 최소 32자** (경고), **64자 권장**. 32자 미만이면 OpenClaw가 경고를 표시하며, 보안 감사에서 CRITICAL로 분류됩니다.
- **allowRequestSessionKey: 절대 허용하지 말 것**. 이 옵션이 `true`이면 외부 요청이 sessionKey를 임의로 지정할 수 있어, 다른 사용자의 세션에 메시지를 주입하는 **세션 인젝션 공격**이 가능합니다.
- **외부 웹훅은 반드시 외부 전용 에이전트로 라우팅**합니다. `agentId`를 `"main"`이 아닌 권한이 제한된 전용 에이전트(예: `"external"`)로 지정하세요.
- **transform 파일 수정 후 Gateway 재시작 필요**. Gateway는 transform 파일을 캐시하므로, 변경 사항을 반영하려면 반드시 `openclaw gateway restart`를 실행하세요.

## 심화: launchd 데몬 설정 (macOS)

Mac에서 Gateway를 시스템 데몬으로 등록하면 부팅 시 자동 시작되고, 크래시 시 자동 재시작됩니다.

```xml
<!-- ~/Library/LaunchAgents/com.openclaw.gateway.plist -->
<plist version="1.0">
<dict>
  <key>Label</key><string>com.openclaw.gateway</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/local/bin/openclaw</string>
    <string>gateway</string>
  </array>
  <key>KeepAlive</key><true/>
  <key>RunAtLoad</key><true/>
</dict>
</plist>
```

### 등록 및 관리

```bash
# 데몬 등록
launchctl load ~/Library/LaunchAgents/com.openclaw.gateway.plist

# 데몬 해제
launchctl unload ~/Library/LaunchAgents/com.openclaw.gateway.plist

# 상태 확인
launchctl list | grep openclaw
```

### 핵심 포인트

- **크래시 시 자동 재시작**: `KeepAlive: true` 설정으로 프로세스가 종료되면 launchd가 즉시 재시작합니다.
- **부팅 시 자동 시작**: `RunAtLoad: true` 설정으로 Mac 부팅(또는 로그인) 시 자동으로 Gateway가 시작됩니다.
- plist 파일 수정 후에는 반드시 `unload` → `load` 순서로 재등록해야 변경 사항이 반영됩니다.
