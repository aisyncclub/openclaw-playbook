# Ch.07 메모리 — 기억의 3단계

> 💡 **코딩 불필요.** 메모리 파일은 마크다운으로 작성합니다.

> "AI는 금붕어처럼 3초 기억력이다?" — 아닙니다. 제대로 설계하면 AI는 당신보다 더 정확하게 기억합니다.

이번 챕터에서는 OpenClaw 에이전트의 **기억 구조**를 해부합니다. 세션이 끝나면 모든 게 사라지는 게 아니라, 기억을 **어디에, 어떻게** 저장하느냐에 따라 에이전트의 역량이 완전히 달라집니다.

---

## 1. RAG vs Full-context Loading

AI에게 정보를 기억시키는 방식은 크게 두 가지입니다.

### RAG (Retrieval-Augmented Generation)

```
질문 → 벡터 검색 → 관련 조각(chunk) 추출 → LLM에 전달
```

- 대규모 문서에서 **관련 부분만** 꺼내서 프롬프트에 넣는 방식
- 장점: 토큰을 아낄 수 있음
- 단점: 검색 품질에 의존, 맥락이 잘릴 수 있음, 인프라 필요 (임베딩 DB, 청킹 파이프라인)

### Full-context Loading (에이전트 방식)

```
파일 통째로 읽기 → 전체를 프롬프트에 넣기
```

- **에이전트는 RAG를 쓰지 않습니다.** 파일을 통째로 읽습니다.
- 장점: 맥락이 잘리지 않음, 구현이 단순, 검색 실패 없음
- 단점: 토큰 소비가 큼

### 왜 에이전트는 RAG를 안 쓸까?

| 기준 | RAG | Full-context |
|------|-----|-------------|
| 구현 복잡도 | 높음 (임베딩, 벡터DB) | 낮음 (파일 읽기) |
| 맥락 보존 | 부분적 | 완전 |
| 비용 | 토큰 절약 | 토큰 많이 씀 |
| 정확도 | 검색 품질에 의존 | 전체가 들어가니 높음 |
| 적합한 상황 | 수백 페이지 문서 | 수십 페이지 이하의 핵심 문서 |

에이전트의 설계 철학은 **"핵심 문서를 정제해서 통째로 넣자"**입니다. AGENTS.md, MEMORY.md 같은 파일은 어차피 수십 KB 수준이라 통째로 넣어도 됩니다. 대신 그 파일 자체를 **잘 정리하는 것**이 핵심입니다.

> **실전 감각**: RAG가 필요한 건 수백 페이지짜리 매뉴얼을 다룰 때입니다. 에이전트의 정체성과 규칙을 다루는 수준이라면 Full-context가 훨씬 낫습니다.

## 2. 기억의 3단계

에이전트의 기억은 **3개 층**으로 이루어져 있습니다. 아래로 갈수록 오래 살아남습니다.

```
┌─────────────────────────────────────────────┐
│  Layer 3: 시스템 파일 (AGENTS.md, MEMORY.md) │  ← 영구 기억
│  ─────────────────────────────────────────── │
│  Layer 2: 메모리 파일 (memory/YYYY-MM-DD.md) │  ← 세션 간 다리
│  ─────────────────────────────────────────── │
│  Layer 1: 대화 맥락 (현재 세션의 대화)        │  ← 가장 휘발적
└─────────────────────────────────────────────┘
```

### Layer 1: 대화 맥락 (세션 안의 기억)

**비유: 칠판**

수업 중에 칠판에 적는 내용입니다. 수업이 끝나면 지워집니다.

- 현재 세션의 모든 대화 내용
- 가장 풍부하지만, 가장 먼저 사라짐
- 컨텍스트 윈도우에 직접 들어있음
- 세션이 끝나면? **증발합니다.**

```
사용자: 오늘 마케팅 회의에서 나온 핵심 3가지 정리해줘
에이전트: (정리함)
사용자: 고마워!
--- 세션 종료 ---
(다음 세션에서 "아까 그 3가지 뭐였지?"라고 물으면... 모릅니다)
```

### Layer 2: 메모리 파일 (세션 사이의 다리)

**비유: 노트**

칠판에 적었던 중요한 내용을 노트에 옮겨 적는 것입니다. 다음 수업 때 노트를 펼치면 됩니다.

- `memory/YYYY-MM-DD.md` 형식의 일별 파일
- 세션에서 발견한 중요한 내용을 여기에 기록
- 다음 세션 시작 시 이 파일을 읽으면 맥락이 이어짐

```markdown
<!-- memory/2026-03-22.md -->
# 2026-03-22 작업 기록

## 마케팅 회의 핵심
1. Q2 타겟: AI 커머스 셀러 교육 과정 런칭
2. 유튜브 콘텐츠 주 2회 → 주 3회로 증가
3. 리드 마그넷: "AI 셀러 체크리스트 PDF" 제작 확정

## 기술 작업
- 데이터 처리 스크립트 v2 완성 (대량 건 자동 분리 로직 추가)
- 버그: 코드값 5자리 미만일 때 0 패딩 안 되는 문제 발견 → 내일 수정
```

### Layer 3: 시스템 파일 (영구 기억)

**비유: 교실 벽에 붙인 쪽지**

칠판은 매번 지워지고, 노트는 잃어버릴 수 있지만, 교실 벽에 붙인 쪽지는 **항상** 거기 있습니다.

- `AGENTS.md` — 에이전트의 정체성, 역할, 규칙
- `MEMORY.md` — 프로젝트 전체에 걸친 핵심 기억
- 매 세션 시작 시 **원본 그대로** 로딩됨
- 컴팩션의 영향을 받지 않음

```markdown
<!-- MEMORY.md -->
# 핵심 기억

## 프로젝트 컨텍스트
- 채널명: AI싱크클럽
- 주요 타겟: 온라인 셀러, 1인 창업자
- 톤: 전문적이지만 친근한 형/누나 느낌

## 기술 스택
- Bun (Node.js 대신)
- Claude API (OpenAI 아님)
- xlsx 처리: SheetJS

## 반복되는 실수 방지
- 스프레드시트 컬럼은 0-indexed (col 10 = K열)
- 보고서: 10건 이상이면 반드시 사용자 확인 먼저
```

## 3. 기억의 승격

기억은 아래에서 위로 **승격**됩니다. 모든 대화 내용이 영구 기억이 될 필요는 없습니다. 정말 중요한 것만 올라갑니다.

```
대화에서 중요한 것 발견
→ memory/ 파일에 기록
→ 정말 핵심이면 MEMORY.md나 AGENTS.md에 반영
```

### 승격 기준

| 단계 | 어디에 저장 | 승격 조건 |
|------|-----------|----------|
| 대화 → memory/ | memory/YYYY-MM-DD.md | "내일도 알아야 할 내용인가?" |
| memory/ → MEMORY.md | MEMORY.md | "매 세션마다 알아야 할 내용인가?" |
| memory/ → AGENTS.md | AGENTS.md | "에이전트의 행동 규칙을 바꿔야 하는가?" |

### 승격 예시

**상황**: 대화 중 사용자가 "주간 리포트에서 10건 이상이면 별도 파일로 분리해"라고 말함.

```
[1단계 - 대화]
사용자: 주간 리포트에서 10건 이상이면 별도 파일로 분리해줘
에이전트: 알겠습니다. 10건 이상은 별도 파일로 분리하겠습니다.

[2단계 - memory/ 기록]
# memory/2026-03-22.md에 추가:
- 주간 리포트: 10건 이상 → 별도 파일 분리 (사용자 요청)

[3단계 - AGENTS.md 반영] (이것이 영구 규칙이 되어야 한다면)
# AGENTS.md의 리포트 섹션에 추가:
- 10건 이상: 별도 파일로 분리 처리
```

> **판단 기준**: "이걸 까먹으면 사용자가 다시 말해야 하나?" — Yes라면 승격하세요.

## 4. 컴팩션과 기억 보호

### 컴팩션이란?

대화가 길어지면 컨텍스트 윈도우가 꽉 찹니다. 이때 오래된 대화를 **요약**해서 압축하는 과정이 컴팩션입니다.

```
[컴팩션 전]
메시지 1: 안녕하세요, 오늘 리포트 작업할게요
메시지 2: PDF로 해주세요
메시지 3: (리포트 생성 과정...)
메시지 4: (대량 건 분리 처리...)
메시지 5: 감사합니다
메시지 6: 이제 데이터 입력할게요
메시지 7: ...

[컴팩션 후]
[요약: 주간 리포트를 생성하고 대량 건 분리를 완료함]
메시지 6: 이제 데이터 입력할게요
메시지 7: ...
```

### 컴팩션에서 안전한 것 vs 위험한 것

| 구분 | 컴팩션 영향 | 이유 |
|------|-----------|------|
| 시스템 파일 (AGENTS.md, MEMORY.md) | **안전** | 매번 원본 로딩 |
| memory/ 파일 | **안전** | 파일로 저장되어 있음 |
| 오래된 대화 내용 | **위험** | 요약 과정에서 세부사항 유실 |

### 기억 보호 전략

**핵심 원칙: 중요한 건 파일에 적어라.**

컴팩션은 대화(Layer 1)에만 영향을 줍니다. 파일에 기록된 것(Layer 2, 3)은 절대 컴팩션 대상이 아닙니다.

```
[안전하지 않은 패턴]
사용자: "앞으로 리포트는 항상 건수 체크해줘"
→ 대화에만 남아있음
→ 컴팩션되면 사라질 수 있음

[안전한 패턴]
사용자: "앞으로 리포트는 항상 건수 체크해줘"
에이전트: memory/2026-03-22.md에 기록했습니다.
→ 파일에 저장됨
→ 컴팩션과 무관하게 보존
```

## 5. Prompt Caching

### 원리

API를 호출할 때마다 시스템 프롬프트 + 대화 내용을 보냅니다. Prompt Caching은 **변하지 않는 부분**을 캐시에 저장해서 비용을 줄이는 기술입니다.

```
┌──────────────────────────────────┐
│  시스템 프롬프트 (잘 안 바뀜)     │  ← 캐시 대상 (앞쪽)
│  ─────────────────────────────── │
│  AGENTS.md 내용                  │  ← 캐시 대상
│  MEMORY.md 내용                  │  ← 캐시 대상
│  ─────────────────────────────── │
│  대화 메시지들 (매번 바뀜)        │  ← 캐시 안 됨 (뒤쪽)
└──────────────────────────────────┘
```

### 배치 순서가 중요하다

```
[올바른 순서]
1. 시스템 프롬프트    ← 캐시 O (거의 안 변함)
2. AGENTS.md         ← 캐시 O (거의 안 변함)
3. MEMORY.md         ← 캐시 O (가끔 변함)
4. memory/ 파일들     ← 캐시 △ (날마다 변함)
5. 대화 메시지들      ← 캐시 X (매번 변함)

[잘못된 순서]
1. 대화 메시지들      ← 매번 변하니까...
2. 시스템 프롬프트    ← 앞이 바뀌면 캐시 무효!
```

### 비용 절감 효과

```
일반 입력 토큰 비용:     $3 / 1M tokens (Claude Sonnet 기준)
캐시 적중 시:           $0.3 / 1M tokens
                        → 90% 절감!

캐시 미스 (최초 저장):   $3.75 / 1M tokens (25% 추가)
```

> **실전 감각**: AGENTS.md와 MEMORY.md를 합쳐서 5,000 토큰이라면, 100번 대화하면 약 $1.35 절약됩니다. 소소해 보이지만 쌓이면 큽니다.

## 6. 세션의 일생

하나의 세션이 태어나서 죽기까지의 전체 생애주기입니다.

```
[1. 탄생]
시스템 파일 로딩: AGENTS.md, MEMORY.md, TOOLS.md
→ "나는 누구인가, 무엇을 할 수 있는가"를 인식

[2. 기억 복구]
memory/ 폴더의 최근 파일 로딩
→ "최근에 무슨 일이 있었는가"를 파악

[3. 대화 축적]
사용자와 대화 시작
→ 컨텍스트 윈도우에 대화 내용이 쌓임

[4. 컴팩션 (필요시)]
컨텍스트 윈도우가 한계에 가까워지면
→ 오래된 대화를 요약/압축
→ 세부사항 일부 유실 가능

[5. 기억 전이]
중요한 발견이 있으면
→ memory/YYYY-MM-DD.md에 기록
→ 정말 핵심이면 MEMORY.md에 반영

[6. 종료]
세션 종료
→ 대화 내용(Layer 1)은 사라짐
→ 파일에 기록된 것(Layer 2, 3)은 남음
```

### 시각적 타임라인

```
시간 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

세션 시작        대화 중           컴팩션          세션 종료
    |              |                |               |
    ├─ 시스템 파일  ├─ 대화 축적     ├─ 오래된 대화   ├─ 대화 소멸
    │  로딩        │               │  요약          │
    ├─ memory/     ├─ 중요 발견!    │               ├─ memory/ 보존
    │  파일 로딩   │  → 파일 기록   │               │
    │              │               │               ├─ 시스템 파일
    │              │               │               │  변함없이 보존
```

## 7. MEMORY.md 작성 가이드

### 뭘 적어야 하는가?

**적어야 하는 것:**
- 프로젝트의 핵심 맥락 (이름, 목적, 타겟)
- 기술 스택과 제약사항
- 반복해서 발생하는 실수와 방지법
- 사용자의 선호/스타일
- 자주 쓰는 경로와 파일 위치

**적지 말아야 하는 것:**
- 일시적인 작업 내용 (→ memory/ 파일에)
- 너무 세부적인 코드 로직 (→ 코드 주석에)
- 자주 바뀌는 정보 (→ 캐시 효율 저하)

### MEMORY.md 작성 원칙

```markdown
# 좋은 예: 간결하고 행동 지향적
- 보고서: 10건 이상이면 반드시 사용자 확인 먼저
- 파일 저장 시 인코딩: UTF-8 (EUC-KR 절대 사용 금지)

# 나쁜 예: 장황하고 서술적
- 2026년 3월 15일에 보고서를 작업했는데, 10건 이상인 경우가
  있었습니다. 이때 사용자가 확인을 원했기 때문에 앞으로는...
```

## 9. memory/ 폴더 운영법

### 파일 명명 규칙

```
memory/
├── 2026-03-20.md
├── 2026-03-21.md
├── 2026-03-22.md    ← 오늘
└── archive/         ← 오래된 파일 보관
    ├── 2026-02-*.md
    └── ...
```

### 일별 파일 구조

```markdown
<!-- memory/2026-03-22.md -->
# 2026-03-22

## 작업 요약
- [완료] 주간 리포트 생성 (15건)
- [완료] 데이터 입력 (32건)
- [진행중] 유튜브 스크립트 초안

## 발견/결정사항
- 신규 카테고리 패턴 추가 필요 (기존 패턴에 없음)
- 사용자 요청: 리포트 파일명에 건수 포함하기로 함

## 내일 할 일
- 유튜브 스크립트 마무리
- 신규 카테고리 패턴 AGENTS.md에 추가

## 이슈/버그
- 코드값 0 패딩 버그 아직 미해결
```

### 정리 주기

```
매일:     오늘의 memory/ 파일 작성
매주:     지난주 파일 중 핵심만 MEMORY.md에 승격
매월:     2개월 이상 된 파일 → archive/ 이동
분기별:   MEMORY.md 전체 리뷰 및 정리
```

> **팁**: memory/ 파일이 30개 이상 쌓이면 세션 시작 시 로딩 시간이 늘어납니다. 최근 7일치만 로딩하고, 나머지는 필요할 때 참조하는 방식을 권장합니다.

## 10. 실전 팁 3가지

### 팁 1: "기록 트리거"를 만들어라

대화 중 특정 패턴이 나오면 자동으로 기록하도록 AGENTS.md에 규칙을 넣으세요.

```markdown
<!-- AGENTS.md에 추가 -->
## 기록 트리거
다음 상황이 발생하면 반드시 memory/ 파일에 기록할 것:
- 사용자가 "앞으로는~" 또는 "항상~"이라고 말할 때
- 새로운 버그를 발견했을 때
- 기존 규칙을 변경하는 결정이 내려졌을 때
- 새로운 데이터 패턴이나 클라이언트가 추가될 때
```

### 팁 2: "기억 테스트"를 해라

새 세션을 시작할 때, 이전 세션의 핵심 내용을 에이전트에게 물어보세요.

```
사용자: 어제 작업한 내용 요약해줘
에이전트: (memory/2026-03-21.md를 읽고 답변)

→ 대답이 정확하면: 기억 시스템이 잘 작동 중
→ 대답이 부실하면: memory/ 파일 작성이 부족한 것
```

### 팁 3: "MEMORY.md는 이력서처럼" 써라

MEMORY.md는 에이전트의 이력서입니다. 새로운 세션이 시작될 때마다 이 파일을 읽고 "나는 이런 프로젝트에서 이런 역할을 하는 에이전트구나"를 파악합니다.

```markdown
<!-- 나쁜 MEMORY.md: 일기장처럼 씀 -->
3월 20일에 리포트 만들고, 3월 21일에 데이터 입력했고...

<!-- 좋은 MEMORY.md: 이력서처럼 씀 -->
## 핵심 역할
- 데이터 관리 시스템 운영 (리포트 생성, 데이터 입력)
- 유튜브 콘텐츠 기획 보조

## 절대 규칙
- 보고서 10건+ → 사용자 확인 필수
- 파일 인코딩: UTF-8 고정
```

## 심화: 컨텍스트 프루닝과 캐시

대화가 길어지면 컨텍스트 윈도우가 부족해집니다. `openclaw.json`에서 프루닝(정리)과 컴팩션(압축) 전략을 세밀하게 제어할 수 있습니다.

```json5
agents: {
  defaults: {
    contextPruning: { mode: "cache-ttl", ttl: "1h" },
    compaction: { mode: "safeguard" }
  }
}
```

### contextPruning: cache-ttl 모드

`cache-ttl` 모드는 **1시간 동안 참조되지 않은 컨텍스트를 자동으로 정리**합니다.

```
09:00 — 사용자: "리포트 만들어줘" → 컨텍스트에 추가
09:30 — 사용자: "건수 확인해줘"  → 09:00 메시지 참조됨 (TTL 갱신)
10:00 — 사용자: "데이터 입력해줘"  → 새로운 주제 시작
11:00 — 프루닝 실행 → 09:00, 09:30 메시지 정리 (1시간 동안 미참조)
```

TTL 값은 업무 성격에 따라 조정합니다:
- `"30m"`: 짧은 대화가 빈번한 경우 (CS 응대)
- `"1h"`: 일반적인 업무 대화 (권장)
- `"4h"`: 긴 호흡의 작업 (코드 리뷰, 문서 작성)

### compaction: safeguard 모드

`safeguard` 컴팩션은 **시스템 프롬프트를 보호하면서 대화만 압축**합니다.

```
[safeguard 컴팩션 동작]
보호 대상 (절대 압축 안 함):
  - 시스템 프롬프트
  - AGENTS.md 내용
  - MEMORY.md 내용
  - TOOLS.md 내용

압축 대상:
  - 오래된 대화 메시지 → 요약으로 대체
  - 도구 실행 결과 → 핵심 결과만 보존
```

다른 컴팩션 모드와의 비교:

| 모드 | 동작 | 위험도 |
|------|------|--------|
| `"none"` | 압축 안 함 (컨텍스트 초과 시 에러) | 안전하지만 제한적 |
| `"aggressive"` | 오래된 것부터 적극 압축 | 세부사항 유실 위험 |
| `"safeguard"` | 시스템 파일 보호 + 대화만 압축 | 권장 — 균형 잡힌 전략 |

## 심화: Prompt Caching 설정

Prompt Caching은 비용 절감의 핵심입니다. `openclaw.json`에서 모델별로 캐시 전략을 지정할 수 있습니다.

```json5
agents: {
  defaults: {
    models: {
      "anthropic/claude-opus-4-6": {
        params: { cacheRetention: "long" }  // none | short (5min) | long (1h)
      }
    }
  }
}
```

### cacheRetention 옵션

| 값 | 캐시 TTL | 적합한 상황 |
|---|----------|-----------|
| `"none"` | 캐시 없음 | 매번 다른 시스템 프롬프트를 사용하는 경우 |
| `"short"` | 5분 | 빠른 대화 (기본값) |
| `"long"` | 1시간 | 긴 작업 세션, heartbeat 조합 (권장) |

`"long"`을 권장하는 이유: 55분 heartbeat와 조합하면 캐시가 만료되기 직전에 갱신되어, **시스템 프롬프트 비용을 거의 0에 가깝게** 유지할 수 있습니다.

### 캐시 진단

캐시가 제대로 작동하는지 확인하려면 진단 로그를 활성화합니다.

```json5
diagnostics: { cacheTrace: { enabled: true, filePath: "~/.openclaw/logs/cache-trace.jsonl" } }
```

또는 환경변수로 간단히 활성화:

```bash
OPENCLAW_CACHE_TRACE=1
```

진단 로그 출력 예시:

```jsonl
{"ts":"2026-03-23T09:00:00Z","model":"claude-opus-4-6","cacheHit":true,"cachedTokens":8500,"newTokens":1200}
{"ts":"2026-03-23T09:55:00Z","model":"claude-opus-4-6","cacheHit":true,"cachedTokens":8500,"newTokens":800}
{"ts":"2026-03-23T11:05:00Z","model":"claude-opus-4-6","cacheHit":false,"cachedTokens":0,"newTokens":9700}
```

`cacheHit: false`가 빈번하면 캐시 전략을 점검해야 합니다. 시스템 프롬프트가 자주 변경되거나, TTL이 너무 짧은 것이 원인일 수 있습니다.

### Provider별 Prompt Caching 지원

| Provider | 지원 여부 | 비고 |
|----------|:---------:|------|
| Anthropic 직접 | O | 완전 지원. `cacheRetention` 설정 가능 |
| Bedrock Claude | O | AWS Bedrock을 통한 Claude 호출도 캐시 지원 |
| OpenRouter Claude | O | OpenRouter 경유 시에도 캐시 적용 |
| OpenAI 호환 API | X | Prompt Caching 미지원 |
| 기타 Provider | X | 제공자별 확인 필요 |

캐시를 지원하지 않는 Provider를 사용하면 `cacheRetention` 설정은 무시되고, 매 요청마다 전체 토큰 비용이 발생합니다.

## 심화: bootstrapMaxChars 제한

에이전트가 세션을 시작할 때 로드하는 파일(AGENTS.md, MEMORY.md, SOUL.md 등)에는 크기 제한이 있습니다.

```
bootstrapMaxChars:      20,000  (파일 1개당 최대 문자 수)
bootstrapTotalMaxChars: 150,000 (모든 부트스트랩 파일의 합계 최대)
```

### 파일별 제한: 20,000자

하나의 파일이 20,000자를 초과하면 **잘립니다(truncated)**. 잘림이 발생하면 파일 끝부분의 내용이 누락되므로, 중요한 규칙이 파일 뒷부분에 있으면 에이전트가 인식하지 못할 수 있습니다.

```
AGENTS.md (18,000자) → 전체 로드 O
MEMORY.md (25,000자) → 20,000자까지만 로드, 5,000자 잘림!
```

### 전체 합계 제한: 150,000자

모든 부트스트랩 파일의 합계가 150,000자를 초과하면 우선순위가 낮은 파일부터 잘립니다.

### 실전 가이드: 간결하게 유지

| 파일 | 권장 크기 | 내용 |
|------|----------|------|
| SOUL.md | 3,000자 이내 | 정체성, 톤, 핵심 가치 |
| AGENTS.md | 10,000자 이내 | 역할, 규칙, 도구 사용법 |
| MEMORY.md | 5,000자 이내 | 프로젝트 핵심 기억 |
| TOOLS.md | 5,000자 이내 | 도구 매핑, 사용 패턴 |

합계: 약 23,000자. 150,000자 제한에 한참 여유가 있습니다.

**잘림을 방지하는 팁:**
- 장황한 서술 대신 불릿 포인트 사용
- 예시는 최소한으로 (1-2개면 충분)
- 히스토리성 내용은 memory/ 파일로 분리
- "절대 규칙"은 파일 상단에 배치 (잘려도 보존)

> **한 줄 요약**: 대화는 칠판, memory 파일은 노트, AGENTS.md는 교실 벽에 붙인 쪽지. 중요한 건 칠판에서 벽으로 승격시켜라.
