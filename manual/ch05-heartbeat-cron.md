# Ch.05 Heartbeat & Cron — 자동 실행의 기술

> 💡 HEARTBEAT.md 작성은 **코딩 불필요.** Cron 명령어는 복사-붙여넣기로 가능합니다.

> **"에이전트가 알아서 일하게 만드는 건 쉽다. 제때, 적절히, 중복 없이 일하게 만드는 게 어렵다."**

에이전트를 24시간 일하는 비서로 만들고 싶다면, 두 가지 자동 실행 메커니즘을 반드시 이해해야 합니다. **Heartbeat**(심장박동)과 **Cron**(정시 실행). 이 둘을 혼동하면 새벽 3시에 팀원들에게 아침 브리핑이 날아가는 참사가 벌어집니다. (실화입니다.)

---

## 1. Heartbeat란 무엇인가

Heartbeat는 말 그대로 **심장박동**입니다. 일정 간격으로 에이전트가 "깨어나서" 체크리스트를 훑어보고, 할 일이 있으면 하고, 없으면 다시 잠듭니다.

```
[잠듦] → (1시간 경과) → [깨어남] → HEARTBEAT.md 확인 →
  ├─ 할 일 있음 → 실행 → [잠듦]
  └─ 할 일 없음 → "HEARTBEAT_OK" 응답 → [잠듦]
```

핵심은 **에이전트가 스스로 판단한다**는 것입니다. "지금 뭐 할 게 있나?" 체크리스트를 보고, 조건에 맞으면 실행하고 아니면 넘어갑니다.

### Heartbeat의 성격

- 모니터링성 업무에 적합
- "새로운 요청이 들어왔는지 확인해줘"
- "긴급건이 있는지 체크해줘"
- "에러 로그에 이상한 거 없는지 봐줘"

---

## 2. HEARTBEAT.md 작성법

`HEARTBEAT.md`는 에이전트가 깨어날 때마다 읽는 **체크리스트**입니다. 프로젝트 루트 또는 `.claude/` 폴더에 위치합니다.

```markdown
# HEARTBEAT 체크리스트

에이전트가 매 하트비트마다 아래 항목을 순서대로 확인합니다.
할 일이 없으면 `HEARTBEAT_OK`를 응답하고 종료합니다.

---

## 1. 긴급 요청 확인
- Notion "긴급 요청" 데이터베이스에 상태가 "대기중"인 항목이 있는지 확인
- 있으면 → 담당자에게 Slack DM 발송
- 없으면 → 다음 항목으로

## 2. 주문 이상 감지
- 최근 1시간 내 주문 건 중 결제 실패가 3건 이상이면
- → #ops-alert 채널에 경고 메시지 발송
- 3건 미만이면 → 다음 항목으로

## 3. 재고 부족 알림
- 재고가 10개 이하인 상품이 있으면
- → 담당자에게 재고 보충 알림 발송
- 없으면 → HEARTBEAT_OK
```

### 작성 원칙

1. **각 항목은 독립적**이어야 합니다. 1번이 실패해도 2번은 실행되어야 합니다.
2. **조건 → 행동** 구조로 작성합니다. "확인하고, 조건에 맞으면 실행한다."
3. **할 일이 없는 경우**를 반드시 명시합니다. (HEARTBEAT_OK)

---

## 3. openclaw.json Heartbeat 설정

```json
{
  "heartbeat": {
    "every": "1h",
    "activeHours": {
      "start": "08:00",
      "end": "22:00",
      "timezone": "Asia/Seoul"
    }
  }
}
```

| 필드 | 설명 | 예시 |
|------|------|------|
| `every` | 하트비트 간격 | `"30m"`, `"1h"`, `"2h"` |
| `activeHours.start` | 활동 시작 시간 | `"08:00"` |
| `activeHours.end` | 활동 종료 시간 | `"22:00"` |
| `activeHours.timezone` | 타임존 | `"Asia/Seoul"` |

**`activeHours`를 설정하면** 해당 시간대 밖에서는 하트비트가 동작하지 않습니다. 새벽 3시에 깨어나서 아침 브리핑을 보내는 참사를 방지하는 첫 번째 안전장치입니다.

### 간격 설정 가이드

| 업무 성격 | 권장 간격 | 이유 |
|-----------|-----------|------|
| 긴급 모니터링 (장애 감지) | `15m` ~ `30m` | 빠른 대응 필요 |
| 일반 모니터링 (새 요청 확인) | `1h` | 적절한 균형 |
| 느긋한 확인 (재고 체크) | `2h` ~ `4h` | 급하지 않은 업무 |

## 4. Heartbeat vs Cron — 판단 매트릭스

이 둘을 혼동하면 반드시 사고가 납니다. 명확한 판단 기준을 세워둡시다.

### 한 줄 정의

| | Heartbeat | Cron |
|---|-----------|------|
| **성격** | "할 일 있어?" (모니터링) | "지금 해." (정시 실행) |
| **비유** | 당직 근무자가 순찰 도는 것 | 알람 시계 |
| **실행 조건** | 조건이 맞으면 실행 | 시간이 되면 무조건 실행 |
| **빈도** | 매 간격마다 체크 | 정해진 시간에 1회 |

### 판단 매트릭스

| 이런 업무라면... | Heartbeat | Cron |
|-----------------|:---------:|:----:|
| "긴급건 있어?" | O | X |
| "새 요청 들어왔어?" | O | X |
| "에러 로그 이상 없어?" | O | X |
| "매일 9시에 브리핑 보내줘" | X | O |
| "매주 월요일에 주간 리포트" | X | O |
| "매월 1일에 정산 보고서" | X | O |

### 쉬운 구분법

> **질문이 "~있어?"로 끝나면 Heartbeat, "~해줘"로 끝나면 Cron.**
>
> - "새 주문 들어왔**어**?" → Heartbeat
> - "아침 브리핑 보내**줘**" → Cron

## 5. 실전 사고 사례 3가지

실제 OpenClaw 에이전트를 운영하면서 겪은 사고들입니다. 여러분은 같은 실수를 반복하지 마세요.

### 사고 #1: 새벽 3시 아침 브리핑 발송

**상황**: "매일 아침 브리핑을 보내줘"라고 HEARTBEAT.md에 적었습니다.

```markdown
# BAD - 이렇게 쓰면 안 됩니다

## 매일 아침 브리핑
- 3개 사업 현황을 종합하여 브리핑 생성
- Slack #morning-briefing 채널에 발송
```

**결과**: 에이전트는 새벽 3시 하트비트에서 "아침 브리핑이라고 되어 있네? 보내야지"라고 판단. 팀원들이 새벽에 알림을 받았습니다.

**원인**: **시간 조건이 명시되지 않았습니다.** 에이전트는 "아침"이라는 단어를 사람처럼 해석하지 않습니다. 명시적인 시간 범위가 없으면 조건이 충족된 것으로 판단합니다.

**해결책**:

```markdown
# GOOD - 시간 조건을 반드시 명시

## 매일 오전 9시 — 아침 브리핑
> **시간 조건**: KST 08:30~09:30 사이 하트비트에서만 실행.
> 이 시간대 밖이면 절대 발송 금지!

- 3개 사업 현황을 종합하여 브리핑 생성
- Slack #morning-briefing 채널에 발송
```

> **절대 규칙 #1**: HEARTBEAT.md에 "발송", "알림", "메시지"가 포함된 항목에는 반드시 **시간 조건**을 명시한다.

---

### 사고 #2: 아침 브리핑 2번 발송 (중복)

**상황**: 아침 브리핑을 HEARTBEAT.md에도 적고, Cron으로도 등록했습니다.

```markdown
# HEARTBEAT.md
## 매일 오전 9시 — 아침 브리핑
> **시간 조건**: KST 08:30~09:30 사이 하트비트에서만 실행.
- 브리핑 생성 후 Slack 발송
```

```bash
# 동시에 이것도 등록
openclaw cron add \
  --name "아침 브리핑" \
  --cron "0 9 * * *" \
  --tz "Asia/Seoul" \
  --message "팀에게 아침 브리핑 발송"
```

**결과**: 9시에 Cron이 발송하고, 9시 하트비트에서도 발송. 팀원들이 동일한 브리핑을 2번 받았습니다.

**원인**: **같은 작업이 Heartbeat과 Cron 양쪽에 등록되었습니다.**

**해결책**: 하나의 작업은 반드시 **한 곳에만** 등록합니다.

```
정시 발송 업무 → Cron에만 등록 (HEARTBEAT.md에서 제거)
모니터링 업무 → HEARTBEAT.md에만 등록 (Cron 등록하지 않음)
```

> **절대 규칙 #2**: 하나의 작업은 Heartbeat 또는 Cron 중 **한 곳에만** 등록한다. 양쪽 모두에 등록하면 100% 중복 실행된다.

---

### 사고 #3: 같은 오류 15번 보고 (메시지 폭탄)

**상황**: "에러 로그에 이상이 있으면 Slack에 알려줘"를 HEARTBEAT.md에 등록했습니다.

```markdown
## 에러 모니터링
- 에러 로그에 새로운 에러가 있으면 Slack #ops-alert에 발송
```

**결과**: 서버에 에러가 1건 발생. 이후 매 하트비트(1시간)마다 같은 에러를 계속 보고. 하루에 15번, 이틀간 30번의 동일한 알림이 발송되었습니다.

**원인**: **"이미 보고한 건을 다시 보고하지 마라"는 조건이 없었습니다.** 에이전트 입장에서는 매번 깨어날 때마다 에러 로그를 확인하고, 에러가 있으니 보고하는 것이 당연합니다.

**해결책**:

```markdown
## 에러 모니터링
- 에러 로그에 **새로운** 에러가 있으면 Slack #ops-alert에 발송
- **중복 방지**: 이미 보고한 에러와 동일한 내용이면 발송하지 않음
- **판단 기준**: 에러 메시지 + 발생 시간이 이전 보고 건과 다를 때만 발송
- **최대 빈도**: 동일 유형의 에러는 최초 보고 후 6시간 이내 재보고 금지
```

> **절대 규칙 #3**: 알림/발송 항목에는 반드시 **중복 방지 조건**과 **최대 빈도 제한**을 명시한다.

## 6. HEARTBEAT.md 시간 조건 필수 표기법

사고 #1에서 배운 교훈을 표준 표기법으로 정리합니다. 발송/알림이 포함된 모든 항목은 아래 형식을 따릅니다.

```markdown
## {업무 이름}
> **시간 조건**: KST {시작}~{끝} 사이 하트비트에서만 실행.
> 이 시간대 밖이면 절대 발송 금지!

- 실행할 내용
```

### 예시 모음

```markdown
## 매일 오전 9시 — 아침 브리핑
> **시간 조건**: KST 08:30~09:30 사이 하트비트에서만 실행.
> 이 시간대 밖이면 절대 발송 금지!

- 3개 사업 현황을 종합하여 브리핑 생성
- Slack #morning-briefing 채널에 발송

---

## 매일 오후 6시 — 일일 리포트
> **시간 조건**: KST 17:30~18:30 사이 하트비트에서만 실행.
> 이 시간대 밖이면 절대 발송 금지!

- 오늘 처리된 주문 건수, 매출, 이슈 요약
- Slack #daily-report 채널에 발송
```

### 왜 범위(08:30~09:30)로 쓰는가?

하트비트가 정확히 9:00에 뛰는 보장이 없습니다. `every: "1h"`로 설정하면 8:17, 9:17처럼 뛸 수도 있습니다. 따라서 **정시가 아니라 범위**로 지정해야 합니다.

> 만약 정확히 9:00에 실행해야 한다면? → **Cron을 사용하세요.** 그게 Cron의 존재 이유입니다.

## 7. Cron 설정

정시에 정확히 실행해야 하는 업무는 Cron을 사용합니다.

### cron add 명령어

```bash
openclaw cron add \
  --name "아침 브리핑" \
  --cron "0 9 * * *" \
  --tz "Asia/Seoul" \
  --session isolated \
  --message "팀에게 아침 브리핑 발송. 3개 사업 현황을 종합하여 브리핑 생성 후 Slack #morning-briefing에 발송."
```

| 옵션 | 설명 |
|------|------|
| `--name` | 작업 이름 (관리용, 중복 불가) |
| `--cron` | Cron 표현식 |
| `--tz` | 타임존 |
| `--session` | `isolated` (독립 세션) 또는 `shared` (공유 세션) |
| `--message` | 에이전트에게 전달할 메시지 (구체적으로!) |

### Cron 표현식 빠른 참조

```
분  시  일  월  요일
 *   *   *   *   *

0 9 * * *        → 매일 09:00
0 9 * * 1        → 매주 월요일 09:00
0 9 1 * *        → 매월 1일 09:00
0 9,18 * * *     → 매일 09:00, 18:00
0 */2 * * *      → 2시간마다 정각
30 8 * * 1-5     → 평일 08:30
```

### --message를 구체적으로 쓰는 이유

```bash
# BAD - 너무 모호
--message "브리핑 보내줘"

# GOOD - 구체적
--message "3개 사업(커머스, 교육, 컨설팅)의 어제 매출, 오늘 일정, 주요 이슈를 종합하여 브리핑을 생성하고, Slack #morning-briefing 채널에 발송. 형식은 마크다운 테이블."
```

Cron은 독립 세션(`--session isolated`)에서 실행되므로, 이전 대화 맥락이 없습니다. 메시지만으로 에이전트가 완전히 이해할 수 있을 만큼 구체적으로 작성해야 합니다.

### 등록된 Cron 확인 및 관리

```bash
# 목록 확인
openclaw cron list

# 출력 예시:
# NAME             SCHEDULE      TZ           NEXT RUN
# 아침 브리핑       0 9 * * *    Asia/Seoul   2026-03-23 09:00
# 주간 리포트       0 10 * * 1   Asia/Seoul   2026-03-24 10:00

# 삭제
openclaw cron delete --name "아침 브리핑"
```

## 8. HEARTBEAT_OK — 할 일 없으면 OK

에이전트가 하트비트 체크리스트를 전부 확인했는데 할 일이 없으면, **`HEARTBEAT_OK`를 응답하고 종료**합니다.

이것이 중요한 이유:

1. **불필요한 작업 방지**: 할 일이 없는데 억지로 뭔가 하려는 것을 막습니다.
2. **로그 추적**: HEARTBEAT_OK 로그를 보면 "에이전트가 정상적으로 깨어났고, 확인은 했지만 할 일이 없었다"는 것을 알 수 있습니다.
3. **비용 절감**: 불필요한 API 호출이나 메시지 발송을 막습니다.

```markdown
# HEARTBEAT.md 마지막에 반드시 명시

---

## 할 일이 없을 때
- 위의 모든 항목에서 실행 조건이 충족되지 않으면
- `HEARTBEAT_OK`를 응답하고 종료
- **절대 불필요한 메시지를 발송하지 말 것**
```

## 9. 금지 패턴 리스트

실전에서 발견된 **절대 하면 안 되는 패턴**들입니다.

### 금지 #1: 시간 조건 없는 발송

```markdown
# HEARTBEAT.md에서 절대 금지

## 아침 브리핑
- 브리핑 생성 후 Slack 발송
```

시간 조건 없이 발송 작업을 넣으면, **매 하트비트마다** 발송됩니다.

### 금지 #2: 동일 작업 이중 등록

```
HEARTBEAT.md: "9시에 브리핑 발송"
Cron: "0 9 * * * 브리핑 발송"
```

반드시 중복 발송됩니다. **한 곳에만 등록하세요.**

### 금지 #3: 중복 방지 조건 없는 알림

```markdown
## 에러 알림
- 에러가 있으면 알림 발송
```

에러가 해결될 때까지 매 하트비트마다 같은 알림이 반복됩니다.

### 금지 #4: 모호한 조건

```markdown
## 뭔가 이상하면 알려줘
- 시스템에 이상이 있으면 보고
```

"이상"의 기준이 없습니다. 에이전트가 자의적으로 판단하게 됩니다.

### 금지 #5: Heartbeat에서 무거운 작업 실행

```markdown
## 데이터 전체 백업
- 전체 데이터베이스를 백업
```

하트비트는 가볍게, 빠르게 체크하는 용도입니다. 무거운 작업은 Cron에서 실행하세요.

## 10. 실전 자동화 구조

Heartbeat과 Cron을 조합한 실전 구조입니다.

```
┌──────────────────────────────────────────────────┐
│                  하루의 자동화                       │
├──────────────────────────────────────────────────┤
│                                                    │
│  [Cron] 09:00  아침 브리핑 발송                      │
│  [Cron] 18:00  일일 리포트 발송                      │
│  [Cron] 10:00 (월) 주간 리포트 발송                   │
│  [Cron] 09:00 (1일) 월간 정산 리포트                  │
│                                                    │
│  [Heartbeat] 매 1시간                               │
│    ├─ 긴급 요청 확인                                 │
│    ├─ 주문 이상 감지                                 │
│    ├─ 재고 부족 알림                                 │
│    └─ 에러 로그 모니터링                              │
│                                                    │
│  activeHours: 08:00 ~ 22:00 KST                    │
│  (22시 이후 ~ 08시 이전: 하트비트 정지)                │
│                                                    │
└──────────────────────────────────────────────────┘
```

### 역할 분담 원칙

| 역할 | 담당 | 이유 |
|------|------|------|
| 모니터링 | Heartbeat | 언제 발생할지 모르는 이벤트 감지 |
| 정시 발송 | Cron | 정확한 시간에 1회 실행 |
| 긴급 알림 | Heartbeat | 빠른 감지 필요 (간격 15~30분) |
| 일일/주간/월간 보고 | Cron | 정해진 주기, 정해진 시간 |

## 12. 요약: Heartbeat & Cron 절대 규칙

| # | 규칙 | 위반 시 결과 |
|---|------|-------------|
| 1 | 발송/알림 항목에는 **시간 조건** 필수 | 새벽 발송 사고 |
| 2 | 하나의 작업은 **한 곳에만** 등록 | 중복 발송 |
| 3 | 알림에는 **중복 방지 조건** 필수 | 메시지 폭탄 |
| 4 | Heartbeat은 **모니터링**, Cron은 **정시 발송** | 역할 혼선 |
| 5 | activeHours로 **비활동 시간** 설정 | 새벽 동작 |
| 6 | 할 일 없으면 **HEARTBEAT_OK** | 불필요한 동작 |
| 7 | Cron의 --message는 **구체적으로** | 엉뚱한 실행 |

## 심화: Heartbeat 프로덕션 설정

기본 설정을 넘어서, 프로덕션 환경에서 Heartbeat를 세밀하게 제어하는 방법입니다.

```json5
heartbeat: {
  every: "30m",
  model: "anthropic/claude-sonnet-4-6",  // 전용 모델 지정 가능
  target: "last",           // last | none | <channel>
  lightContext: false,       // true = HEARTBEAT.md만 로드
  isolatedSession: false,    // true = 별도 세션에서 실행
  activeHours: { start: "09:00", end: "22:00", timezone: "Asia/Seoul" },
  ackMaxChars: 300,          // HEARTBEAT_OK 최대 길이
  directPolicy: "allow"
}
```

### target 옵션

| 값 | 동작 |
|---|------|
| `"last"` | 마지막으로 활성화된 채널에서 실행. 사용자가 가장 최근에 대화한 채널의 맥락을 이어받음 |
| `"none"` | 내부에서만 실행. 외부 채널에 메시지를 보내지 않고 체크만 수행 |
| `"#ops-alert"` | 특정 채널명을 직접 지정. 항상 해당 채널에서 실행 |

`target: "none"`은 외부 알림 없이 **내부 상태 점검만** 수행할 때 유용합니다. 예를 들어 에러 로그를 모니터링하되, 이상이 발견될 때만 별도 스킬을 호출하여 알림을 보내는 패턴에 적합합니다.

### lightContext: 비용 절감 모드

`lightContext: true`로 설정하면 하트비트 실행 시 **HEARTBEAT.md만** 로드합니다.

```
lightContext: false (기본)
→ 로드: AGENTS.md + MEMORY.md + TOOLS.md + HEARTBEAT.md + memory/ 파일들
→ 약 100K 토큰

lightContext: true
→ 로드: HEARTBEAT.md만
→ 약 2-5K 토큰
```

비용 차이가 극적입니다. 단순 모니터링(API 상태 체크, 에러 로그 확인 등)이라면 `lightContext: true`가 훨씬 효율적입니다. 다만 에이전트가 프로젝트 맥락을 모른 채 판단해야 하므로, HEARTBEAT.md에 필요한 맥락을 모두 포함시켜야 합니다.

### HEARTBEAT_OK와 ackMaxChars

에이전트가 할 일이 없을 때 반환하는 `HEARTBEAT_OK` 응답의 최대 길이를 `ackMaxChars: 300`으로 제한합니다. 이 응답은 자동으로 제거되어 채널을 오염시키지 않습니다.

만약 에이전트가 300자를 초과하는 응답을 하면? 그건 "할 일이 없다"가 아니라 "뭔가 보고할 게 있다"는 뜻이므로, HEARTBEAT_OK로 처리되지 않고 정상 메시지로 남습니다.

### 55분 Heartbeat와 Prompt Caching 최적화

Anthropic의 Prompt Cache TTL은 **1시간**입니다. 하트비트 간격을 `"55m"`으로 설정하면:

```
09:00 — 하트비트 실행 → 시스템 프롬프트 캐시 생성 (TTL: 1h)
09:55 — 하트비트 실행 → 캐시 HIT (비용 90% 절감) + 캐시 TTL 갱신
10:50 — 하트비트 실행 → 캐시 HIT + TTL 갱신
...
```

만약 `"65m"`으로 설정하면 매번 캐시가 만료된 후 실행되어 **캐시 MISS**가 발생합니다. 55분은 캐시 갱신과 비용 절감의 최적 균형점입니다.

## 심화: Cron 고급 기능

기본 `cron add` 명령을 넘어서, 프로덕션에서 필요한 고급 스케줄링 기능들입니다.

### 기본 스케줄링, 1회성 리마인드, 시스템 이벤트

```bash
# 기본 스케줄링
openclaw cron add --name "아침 브리핑" --cron "0 9 * * *" --tz "Asia/Seoul" --session isolated --message "브리핑 생성"

# 1회성 리마인드
openclaw cron add --name "회의 알림" --at "2026-03-24T14:00" --message "회의 시작 10분 전"

# 시스템 이벤트 (메인 세션에 주입)
openclaw system event --text "긴급 확인" --mode now
openclaw system event --text "다음 체크" --mode next-heartbeat
```

### isolated vs main session cron

| 모드 | 동작 | 적합한 작업 |
|------|------|-----------|
| `--session isolated` | 독립 세션에서 실행. 이전 대화 맥락 없음 | 정형화된 작업 (브리핑, 리포트) |
| `--session main` | 메인 세션에 메시지 주입. 기존 맥락 유지 | 맥락이 필요한 확인/판단 작업 |

`isolated`는 매번 깨끗한 상태에서 시작하므로 결과가 일관됩니다. `main`은 현재 대화 맥락을 활용할 수 있지만, 사용자가 대화 중일 때 cron 메시지가 끼어들 수 있습니다.

### 6-field cron (초 단위 지원)

OpenClaw는 표준 5-field cron 외에 **6-field cron**(초 단위)을 지원합니다.

```
초  분  시  일  월  요일
 *   *   *   *   *   *

*/30 * * * * *     → 30초마다
0    0 9 * * *     → 매일 09:00:00 (정확히 0초)
15   0 * * * *     → 매 정시 15초
```

주의: 초 단위 스케줄은 비용이 급증할 수 있습니다. 30초마다 실행하면 하루 2,880회 호출입니다. 반드시 `lightContext: true`와 함께 사용하세요.

### 로드 분산

여러 cron 작업이 같은 시간(예: 매일 9시)에 등록되어 있으면, OpenClaw는 자동으로 **0~5분 사이의 랜덤 지연**을 추가하여 로드를 분산합니다.

```
등록: "0 9 * * *" (아침 브리핑)
등록: "0 9 * * *" (일일 체크)
등록: "0 9 * * *" (재고 확인)

실제 실행:
  아침 브리핑 → 09:00:00
  일일 체크   → 09:02:17 (자동 지연)
  재고 확인   → 09:04:33 (자동 지연)
```

### sessionRetention과 maxConcurrentRuns

```json5
cron: {
  sessionRetention: "24h",    // 크론 세션 이력 24시간 보관
  maxConcurrentRuns: 2         // 같은 cron job의 동시 실행 최대 2개
}
```

- `sessionRetention: "24h"`: 크론 실행 이력을 24시간 동안 보관합니다. `openclaw cron logs --name "아침 브리핑"`으로 최근 실행 결과를 조회할 수 있습니다.
- `maxConcurrentRuns: 2`: 동일 크론 작업이 아직 실행 중인데 다음 스케줄이 도래하면, 최대 2개까지만 동시 실행을 허용합니다. 초과분은 스킵됩니다. 이전 작업이 지연되어 중복 실행되는 사고를 방지합니다.

## 심화: 비용 최적화 패턴

Heartbeat과 Cron의 비용을 최소화하는 실전 패턴들입니다.

### 패턴 1: Heartbeat를 로컬 Ollama 모델로 라우팅

단순 모니터링(API 상태 체크, 파일 존재 확인 등)에는 고가의 Claude 모델이 필요 없습니다. 로컬 Ollama 모델로 라우팅하면 월 약 $50를 절감할 수 있습니다.

```json5
heartbeat: {
  every: "30m",
  model: "ollama/llama3.2:3b",   // 로컬 모델 → API 비용 $0
  lightContext: true,
  activeHours: { start: "09:00", end: "22:00", timezone: "Asia/Seoul" }
}
```

로컬 모델의 한계:
- 복잡한 판단이 필요한 체크리스트에는 부적합
- HEARTBEAT.md를 더 구체적이고 단순하게 작성해야 함
- "이상 여부"를 판단하는 건 어렵지만, "수치가 X 이상인가?"는 가능

### 패턴 2: lightContext + isolatedSession 조합

```json5
heartbeat: {
  lightContext: true,        // HEARTBEAT.md만 로드
  isolatedSession: true,     // 별도 세션에서 실행 (메인 세션 오염 방지)
}
```

이 조합은 **최소 비용 체크**를 구현합니다:
- 로드되는 토큰: HEARTBEAT.md의 크기만큼 (보통 2-5K)
- 메인 세션의 컨텍스트를 전혀 건드리지 않음
- 단점: 프로젝트 맥락을 모르므로, HEARTBEAT.md에 모든 판단 기준을 명시해야 함

### 패턴 3: 여러 cron 대신 heartbeat 배치 체크

```
[비효율적]
cron: "0 * * * *" → API 상태 체크
cron: "0 * * * *" → 에러 로그 확인
cron: "0 * * * *" → 재고 확인
→ 매시간 3개 세션 생성, 3배 비용

[효율적]
heartbeat: { every: "1h" }
HEARTBEAT.md:
  1. API 상태 체크
  2. 에러 로그 확인
  3. 재고 확인
→ 매시간 1개 세션, 1/3 비용
```

비슷한 주기의 모니터링 작업은 HEARTBEAT.md에 통합하세요. Cron은 **정시 발송처럼 독립적인 실행이 필요한 작업**에만 사용합니다.
