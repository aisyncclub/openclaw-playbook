# Ch.11 토큰과 비용 — 새는 돈 잡기

> 💡 RTK 설치는 **1줄 명령어.** 비용 개념 이해가 핵심입니다.

> "코드 짜는 토큰보다 읽는 토큰이 더 많다."

AI 에이전트를 실무에 쓰면 가장 먼저 체감되는 것이 **비용**입니다. 이번 장에서는 토큰이 어디서 새는지 파악하고, 실전에서 검증된 절약 전략을 알려드립니다.

---

## 11.1 핵심 인사이트: 읽기 토큰이 더 비싸다

에이전트가 코드를 작성하는 데 쓰는 토큰(출력)보다, 코드를 읽고 분석하는 데 쓰는 토큰(입력)이 훨씬 많습니다.

```
일반적인 에이전트 세션의 토큰 분포:

┌──────────────────────────────────────────┐
│                                          │
│  입력 토큰 (읽기)      ████████████ 80%  │
│                                          │
│  출력 토큰 (쓰기)      ██ 20%            │
│                                          │
└──────────────────────────────────────────┘
```

입력 토큰이 많은 이유:
- CLI 출력을 통째로 읽음
- API 응답 전체를 컨텍스트에 넣음
- 같은 파일을 여러 번 읽음
- 대화 히스토리가 누적됨
- 시스템 프롬프트가 매번 포함됨

## 11.4 RTK (Rust Token Killer)

CLI 출력의 토큰 낭비를 획기적으로 줄여주는 도구입니다.

### 설치

```bash
brew install rtk-ai/tap/rtk
```

### 초기 설정

```bash
# 글로벌 설정 (모든 프로젝트에 적용)
rtk init --global

# 프로젝트별 설정
rtk init
```

### 사용법

기존 명령어 앞에 `rtk`를 붙이면 됩니다:

```bash
# Before: 원본 출력 (2,074자)
git status

# After: RTK 압축 (457자, 78% 절감)
rtk git status
```

### 실제 비교

```bash
# 원본 git status 출력
$ git status
On branch feature/cs-bot
Your branch is ahead of 'origin/feature/cs-bot' by 3 commits.
  (use "git push" to publish your local commits)

Changes staged for commit:
  (use "git restore --staged <file>..." to unstage)
        new file:   agents/cs-external/AGENTS.md
        modified:   openclaw.json

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   transforms/channeltalk-flatten.js

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        learnings/
        tests/

# RTK 처리 후
$ rtk git status
branch: feature/cs-bot (+3 ahead)
staged: +agents/cs-external/AGENTS.md, ~openclaw.json
unstaged: ~transforms/channeltalk-flatten.js
untracked: learnings/, tests/
```

정보는 동일하지만 **토큰 수가 78% 줄었습니다**.

### 절감량 확인

```bash
# 지금까지 절감한 토큰 확인
rtk gain

# 출력 예시:
# Total tokens saved: 1,247,832
# Total commands: 3,456
# Average savings: 72%
# Estimated cost saved: $12.48
```

### RTK AGENTS.md 지시문

에이전트가 RTK를 활용하도록 지시문을 추가합니다:

```markdown
<!-- AGENTS.md에 추가 -->

- `rtk git status` (git status 대신)

- `rtk git log` (git log 대신)

- `curl` / API 호출 (응답 전체가 필요할 수 있음)
```

## 11.5 5가지 추가 절약 전략

### 전략 1: 세션 관리

대화가 길어지면 컨텍스트 비용이 기하급수적으로 증가합니다.

```
규칙:
- 100K 토큰 도달 시: /compact 실행 (히스토리 압축)
- 150K 토큰 도달 시: 새 세션 시작 (진행 상황 todo.md에 기록)
```

`/compact` 명령은 대화 히스토리를 요약하여 토큰을 절감합니다:

```bash
# 현재 토큰 사용량 확인
/status

# 출력 예시:
# Context: 95,234 / 200,000 tokens (47.6%)
# → compact 권장

# 히스토리 압축
/compact

# 결과:
# Before: 95,234 tokens
# After:  23,456 tokens
# Saved:  71,778 tokens (75.4%)
```

### 전략 2: 서브에이전트 위임

메인 에이전트의 컨텍스트를 깨끗하게 유지하면서, 무거운 작업은 서브에이전트에 맡깁니다.

```
메인 에이전트 (Opus 4)
  │
  ├── "파일 A의 구조를 분석해줘" → 서브에이전트 (Sonnet) → 결과만 받기
  ├── "테스트 결과를 요약해줘" → 서브에이전트 (Sonnet) → 결과만 받기
  └── "코드 리뷰해줘"         → 서브에이전트 (Sonnet) → 결과만 받기
```

서브에이전트의 장점:
- **클린 컨텍스트**: 메인의 히스토리 없이 시작
- **저렴한 모델**: Sonnet 기본 사용 (Opus 대비 저렴)
- **병렬 실행**: 독립적인 작업은 동시에 실행 가능

```markdown
<!-- AGENTS.md에 추가 -->

## 서브에이전트 활용 규칙
- 파일 분석, 코드 리뷰, 테스트 실행은 서브에이전트에 위임
- 서브에이전트에는 필요한 컨텍스트만 전달 (전체 히스토리 X)
- 결과는 요약만 받아서 메인 컨텍스트에 추가
```

### 전략 3: 파일 읽기 최적화

파일 전체를 읽지 말고, 필요한 부분만 읽습니다.

```bash
# 나쁜 예: 파일 전체 읽기 (2,000줄 = ~8,000 토큰)
read src/services/order-service.ts

# 좋은 예: 필요한 부분만 읽기 (50줄 = ~200 토큰)
read src/services/order-service.ts --offset 150 --limit 50
```

에이전트 지시문에 명시합니다:

```markdown
## 파일 읽기 규칙
- 파일 전체를 읽기 전에, 필요한 범위를 먼저 파악할 것
- 200줄 이상의 파일은 offset/limit으로 부분 읽기
- 같은 세션에서 이미 읽은 파일은 다시 읽지 않을 것
- 필요하면 파일 구조만 먼저 확인 (head 20줄)
```

### 전략 4: NO_REPLY (무응답 전략)

에이전트가 모든 메시지에 응답할 필요는 없습니다. 무관한 메시지에는 응답하지 않도록 설정합니다.

```markdown
<!-- agents/cs-external/AGENTS.md -->

## NO_REPLY 규칙
다음 메시지에는 응답하지 마세요 (토큰 절약):
- 시스템 알림 메시지 (자동 생성 메시지)
- "감사합니다", "네", "ㅋㅋ" 등 단순 반응
- 이미 해결된 문의에 대한 추가 인사
- 봇이 보낸 메시지 (transform에서 필터링 실패한 경우 2차 방어)

NO_REPLY 판단 시:
1. 응답하지 않음 (토큰 0 소비)
2. 단, 고객이 추가 질문을 한 것으로 판단되면 응답
```

### 전략 5: 시스템 프롬프트 최적화

AGENTS.md가 길면 매 턴마다 그만큼 토큰이 소비됩니다.

```
Before: AGENTS.md 390줄 → ~6,000 토큰 × 매 턴
After:  AGENTS.md 161줄 → ~2,500 토큰 × 매 턴

20턴 세션 기준:
  Before: 6,000 × 20 = 120,000 토큰
  After:  2,500 × 20 =  50,000 토큰
  절감:                  70,000 토큰 (58%)
```

최적화 방법:

```markdown
# Before (비효율적)
## 파일 읽기 규칙
파일을 읽을 때는 반드시 다음 규칙을 따라야 합니다.
첫째, 파일 전체를 읽기 전에 필요한 범위를 먼저 파악해야 합니다.
둘째, 200줄 이상의 파일은 offset과 limit 파라미터를 사용하여
부분적으로만 읽어야 합니다. 전체를 읽으면 토큰이 낭비됩니다.
셋째, 같은 세션에서 이미 읽은 파일은 다시 읽지 않습니다.

# After (효율적)
## 파일 읽기
- 200줄+: offset/limit 사용
- 재읽기 금지 (세션 내)
- 전체 읽기 전 범위 파악
```

의미는 동일하지만 토큰은 절반 이하입니다.

## 11.6 /status 명령어

현재 세션의 토큰 사용량을 확인합니다.

```bash
/status
```

출력 예시:

```
=== Session Status ===

Token Usage:
  Input:    45,234 tokens  ($0.45)
  Output:    8,123 tokens  ($0.24)
  Cache:    12,456 tokens  (cached, $0.00)
  Total:    65,813 tokens  ($0.69)

Context Window:
  Used:     65,813 / 200,000 (32.9%)
  System:   12,000 (18.2%)
  History:  48,000 (72.9%)
  Current:   5,813 ( 8.8%)

Cache Hit Rate: 27.3%

Recommendations:
  ✓ Context usage normal
  ✓ Cache rate could be improved (target: 40%+)
  → Consider /compact at 100K tokens
```

각 항목의 의미:

| 항목 | 설명 | 목표 |
|------|------|------|
| Input tokens | 에이전트가 읽은 토큰 | 최소화 |
| Output tokens | 에이전트가 생성한 토큰 | 필요한 만큼 |
| Cache hit rate | 캐시된 토큰 비율 | 40% 이상 |
| Context usage | 컨텍스트 윈도우 사용률 | 50% 이하 유지 |

## 11.9 비용 모니터링 루틴

### 일간 체크 (1분)

```bash
# 오늘 사용량 확인

# 체크 포인트:
# - Context 50% 이하인가?
```

## 심화: RTK 상세 설정

RTK(Rust Token Killer)를 OpenClaw 및 Claude Code 환경에 완전히 통합하는 방법입니다.

### 설치

```bash
# Homebrew (macOS/Linux)
brew install rtk

# 또는 스크립트 설치
curl -fsSL https://raw.githubusercontent.com/rtk-ai/rtk/refs/heads/master/install.sh | sh
```

### OpenClaw/Claude Code 통합

```bash
# 비대화형 자동 설정 (CI/CD 및 자동화 환경에 적합)
rtk init -g --auto-patch

# 설정 확인
rtk status
```

`rtk init -g --auto-patch`는 다음을 자동으로 수행합니다:
- 글로벌 설정 파일 생성
- Claude Code의 PreToolUse 훅에 RTK 리라이트 스크립트 등록
- `~/.claude/hooks/rtk-rewrite.sh` 파일 생성

### 절감 통계 확인

```bash
# 절감 통계
rtk gain

# ASCII 그래프로 시각화
rtk gain --graph

# 최적화를 놓친 명령 발견
rtk discover
```

`rtk discover`는 RTK 없이 실행된 명령어 중 최적화 가능한 것을 찾아줍니다. 이를 통해 놓치고 있는 절감 기회를 발견할 수 있습니다.

### Hook 동작 원리

RTK는 **PreToolUse 훅**에서 동작합니다. 에이전트가 CLI 명령을 실행하기 전에 RTK가 명령어를 자동으로 치환하여 출력을 압축합니다.

- `~/.claude/hooks/rtk-rewrite.sh`가 훅 스크립트로 등록됨
- 원본 명령어는 변경 없이 실행되고, 출력만 압축됨
- 에이전트가 받는 정보의 의미는 동일하지만 토큰 수가 대폭 감소

### 실제 절감 사례

| 명령어 | 원본 토큰 | RTK 후 토큰 | 절감률 |
|--------|----------|------------|--------|
| `vitest --run` | ~25,000 | ~100 | -99.6% |
| `git status` | ~2,000 | ~820 | -59% |
| `pnpm outdated` | ~8,000 | ~800 | -90% |
| `git diff` | ~5,000 | ~1,500 | -70% |
| `docker ps` | ~1,200 | ~300 | -75% |

## 심화: 종합 비용 최적화 전략

모든 비용 최적화 전략을 한 눈에 보는 종합 테이블입니다.

| 전략 | 절감률 | 방법 |
|------|--------|------|
| 모델 라우팅 (Haiku 기본) | 85-90% | primary: haiku, fallback: sonnet |
| 세션 최적화 | 80% 시작비용 | Lazy-load, isolatedSession |
| 로컬 heartbeat (Ollama) | heartbeat 100% | model: "ollama/llama3.2:3b" |
| Prompt Caching (long) | 반복 90% | cacheRetention: "long" |
| RTK 출력 압축 | CLI 60-90% | brew install rtk |
| 컨텍스트 프루닝 | 가변 | contextPruning.mode: "cache-ttl" |
| 전체 조합 | $1,200→$50/월 | 위 전략 모두 적용 |

### 각 전략 상세

1. **모델 라우팅**: 일상적인 대화와 간단한 작업은 Haiku로 처리하고, 복잡한 판단이 필요할 때만 Sonnet/Opus로 폴백합니다. 전체 비용의 85-90%를 절감할 수 있습니다.

2. **세션 최적화**: `isolatedSession`을 사용하면 불필요한 컨텍스트 로딩 없이 깨끗한 세션을 시작합니다. Lazy-load로 필요한 시점에만 에이전트를 초기화합니다.

3. **로컬 heartbeat**: 단순 스케줄 체크(heartbeat)는 로컬 LLM(Ollama 등)으로 처리합니다. API 비용이 0원이 됩니다.

4. **Prompt Caching**: `cacheRetention: "long"`으로 설정하면 시스템 프롬프트와 반복되는 컨텍스트가 캐시되어 반복 비용이 90% 감소합니다.

5. **RTK 출력 압축**: CLI 출력을 60-90% 압축하여 입력 토큰을 대폭 절감합니다.

6. **컨텍스트 프루닝**: `cache-ttl` 모드로 오래된 컨텍스트를 자동으로 정리하여 누적 비용을 방지합니다.

**전체 조합 적용 시**: 월 $1,200 수준의 비용을 $50 이하로 줄일 수 있습니다. 전략을 하나씩 적용할 때마다 복합적으로 절감 효과가 쌓입니다.
