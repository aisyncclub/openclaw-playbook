# Ch.03 AGENTS.md — 업무 매뉴얼과 절대 규칙

> 💡 기본 내용은 **코딩 불필요.** 심화(exec-approvals)는 개발자용입니다.

> "SOUL.md가 DNA라면, AGENTS.md는 사내 업무 매뉴얼입니다."

---

## AGENTS.md의 역할

| SOUL.md | AGENTS.md |
|---------|-----------|
| 나는 누구인가 | 나는 어떻게 일하는가 |
| 성격, 말투, 경계 | 규칙, 프로세스, 금지사항 |
| 잘 안 바뀜 | 실수할 때마다 업데이트 |

---

## AGENTS.md 구성

### 1. 세션 시작 프로토콜

```markdown
## 세션 시작 시

1. MEMORY.md를 읽고 현재 상태를 파악한다.
2. memory/ 폴더에서 최근 3일 파일을 확인한다.
3. 진행 중인 TODO가 있으면 이어서 처리한다.
```

### 2. 메모리 작성 규칙

```markdown
## 메모리 규칙

### 파일명
memory/YYYY-MM-DD.md (예: memory/2026-03-22.md)

### 기록 기준
- 새로운 결정사항
- 완료된 중요 작업
- 발생한 이슈와 해결 방법
- 다음에 이어서 할 TODO

### 기록하지 않는 것
- 일상적인 인사/잡담
- 이미 시스템 파일에 있는 정보
- 임시 디버깅 로그
```

### 3. 팀원 권한 체계

```markdown
## 권한 체계

### 대표님 (최고 권한)
- 모든 작업 지시 가능
- 절대 규칙 수정 가능
- 외부 발송 승인 권한

### 팀원
- 자기 담당 업무 요청 가능
- 다른 팀원 정보 조회 불가
- 외부 발송 불가 (대표님 승인 필요)
```

### 4. 절대 규칙

이것이 AGENTS.md의 **핵심**입니다.

---

## 절대 규칙 — 실수가 만드는 "다시는 안 해" 규칙

### 흐름

```
실수 발생 → 원인 분석 → 구체적 규칙 작성 → AGENTS.md에 추가 → 다시는 발생 안 함
```

### 실전 사례

#### 사례 1: "아침 브리핑을 새벽에 보내버렸다"

HEARTBEAT.md에 "아침 브리핑을 보내라"만 적혀있고 "몇 시에"가 없었음. 새벽 0시에 아침 브리핑 전송.

**태어난 절대 규칙:**
```markdown
🕐 시간대(Timezone) — 반드시 KST 기준.
아침 브리핑은 KST 08:30~09:30 사이에만 실행.
이 시간대 밖이면 절대 발송 금지!
```

#### 사례 2: "같은 오류를 15번 보고했다"

카카오톡 앱 장애 시 매 하트비트마다 반복 보고. 하루에 15번 동일 알림.

**태어난 절대 규칙:**
```markdown
🚨 시스템 장애 보고는 최초 1회만!
같은 오류는 1회 보고 후, "고쳤다"고 할 때까지 조용히 스킵.
메모리 파일로 "이미 보고했는지" 반드시 확인.
```

#### 사례 3: "시간을 확인 안 하고 답했다"

기억에 의존한 즉답으로 일정 시간을 두 번 연속 틀림.

**태어난 절대 규칙:**
```markdown
🚨 일정/시간/숫자는 반드시 원본 데이터 조회 후 작성.
기억에 의존한 즉답 → 절대 금지.
"1분 늦더라도 정확한 것"이 100배 나음.
```

---

## "알고 있다" ≠ "실행한다"

| 추상적 원칙 (나쁜 예) | 구체적 규칙 (좋은 예) |
|---------------------|---------------------|
| "불필요한 반복을 피하라" | "최초 1회만 + 고쳤다고 할 때까지 스킵 + 메모리에서 확인" |
| "시간을 잘 확인해라" | "일정/숫자는 반드시 원본 조회 후 작성. 기억 의존 즉답 금지" |
| "적절한 시간에 보내라" | "KST 08:30~09:30 사이에만. 밖이면 절대 금지" |

추상적 원칙은 AI가 "해석"해야 하고, 해석은 틀릴 수 있습니다. 구체적 규칙은 해석할 여지 없이 그대로 따르면 됩니다.

---

## 지침 vs 시스템 — 2중 방어선

| 방어 수단 | 방식 | 약점 |
|----------|------|------|
| **지침** (절대 규칙) | "이러면 안 돼" 명시 | AI가 매번 판단해야 함 |
| **시스템** (구조적 방어) | 실행 경로 자체를 제거 | 판단 불필요, 놓칠 수가 없음 |

**실전에서는 둘 다 씁니다:**

```
1차 방어선 — 절대 규칙: "중복 발송하지 마" (AI의 의식적 판단)
2차 방어선 — 시스템: 크론잡으로 이관, HEARTBEAT에서 삭제 (실행 경로 자체가 하나)
```

## 절대 규칙 운영 팁

1. **처음엔 0개로 시작** — 미리 만들지 마세요. 실수가 발생하면 그때 추가합니다.
2. **구체적으로** — "잘 해라" 대신 "이 경우에 이렇게 해라".
3. **이유를 적어라** — 왜 이 규칙이 생겼는지. 나중에 맥락을 잊으면 규칙이 형해화됩니다.
4. **정기 점검** — 월 1회 절대 규칙을 리뷰. 더 이상 유효하지 않은 규칙은 제거.

> AGENTS.md의 절대 규칙은 **조직의 실수 DB**입니다. 실제 운영 사례에서 9일 동안 0개에서 14개로 증가했습니다. 규칙이 많아질수록 에이전트는 더 안정적으로 일합니다.

## 심화: exec-approvals — 실행 권한 3단계

에이전트가 쉘 명령을 실행할 수 있게 하면 강력하지만 위험합니다. OpenClaw는 3단계 보안 모드를 제공합니다.

### 보안 모드

| 모드 | 동작 | 위험도 |
|------|------|--------|
| `deny` | 모든 쉘 실행 차단 | 안전 (메시지 전용 에이전트에 적합) |
| `allowlist` | 허용된 명령만 실행, 나머지는 승인 요청 | 권장 (대부분의 운영 에이전트) |
| `full` | 모든 쉘 실행 허용 | 위험 (개발 전용, 프로덕션 비권장) |

### exec-approvals.json 설정

```json5
// ~/.openclaw/exec-approvals.json
{
  "security": "allowlist",
  "ask": "on-miss",          // on-miss: 허용 목록에 없으면 채팅으로 승인 요청
  "safeBins": [
    "jq", "cut", "head", "tail", "wc",    // stdin-only 도구
    "date", "cal", "echo", "printf",
    // ⚠️ curl, wget은 safeBins에 넣지 말 것 (외부 서버로 데이터 POST 가능)
  ],
  "allowedCommands": [
    "git status", "git log *", "git diff *",
    "ls *", "cat *", "find *"
  ],
  "deniedPatterns": [
    "rm -rf *",
    "sudo *",
    "chmod 777 *",
    "*(>|>>|\\|) /etc/*"
  ]
}
```

### safeBins 개념

`safeBins`는 **stdin으로만 데이터를 받는 읽기 전용 도구**입니다. 파일 시스템을 변경하지 않으므로 승인 없이 실행할 수 있습니다.

```
안전: jq, cut, head, tail, wc, sort, uniq, grep, awk, sed (파이프 입력만)
주의: curl, wget (외부 서버로 데이터 전송 가능 — safeBins에 넣지 말 것)
위험: rm, mv, cp, chmod, chown, dd, mkfs
```

> **절대 safeBins에 넣지 말 것**: `python`, `node`, `bash`, `sh`, `bun`, `deno` 등 인터프리터. 인터프리터는 어떤 코드든 실행할 수 있으므로 safeBins에 넣으면 보안이 무력화됩니다.

### 채팅 승인 명령

`ask: "on-miss"` 설정 시 허용 목록에 없는 명령을 실행하려 하면 에이전트가 채팅으로 승인을 요청합니다.

```
🔒 실행 승인 요청 [id: exec-a7f3]
명령: python3 scripts/migrate.py --dry-run
에이전트: main
승인하려면: /approve exec-a7f3 allow-once
```

승인 옵션:

| 명령 | 동작 |
|------|------|
| `/approve <id> allow-once` | 이번 한 번만 허용 |
| `/approve <id> allow-always` | 이 명령을 영구 허용 목록에 추가 |
| `/approve <id> deny` | 거부 |

### 쉘 체이닝 규칙

`&&`, `||`, `;`로 연결된 명령은 **각각 개별 검사**됩니다.

```bash
# 이 명령은 두 개로 분리되어 각각 검사됨
git status && rm -rf .git
#  → git status: 허용
#  → rm -rf .git: 거부 → 전체 명령 차단
```

파이프(`|`)로 연결된 명령도 마찬가지입니다. 체인 중 하나라도 거부되면 **전체가 차단**됩니다.

## 심화: 도구 권한 프로파일

에이전트마다 사용할 수 있는 도구의 범위가 다릅니다. `tools.profile`로 프리셋을 지정하고, `deny`로 세부 조정합니다.

### 프로파일 3종

```json5
tools: {
  profile: "messaging",  // messaging | coding | full
  deny: ["group:automation", "sessions_spawn"],
  exec: { security: "allowlist", ask: "on-miss" },
  fs: { workspaceOnly: true }
}
```

| 프로파일 | 포함 도구 | 적합한 에이전트 |
|---------|----------|---------------|
| `messaging` | 메시지 전송/읽기, 리액션, 파일 첨부 | 외부 고객 응대 에이전트, CS봇 |
| `coding` | messaging + 파일 읽기/쓰기 + 쉘 실행 + git | 개발 에이전트, 분석봇 |
| `full` | coding + 에이전트 간 통신 + 크론 관리 + 세션 생성 | 팀장 에이전트, 오케스트레이터 |

### 프로파일별 도구 목록 상세

**messaging** (최소 권한):
- `messages_send`, `messages_read`, `reactions_add`, `files_upload`
- `memory_read`, `memory_write`
- 쉘 실행 불가, 파일시스템 직접 접근 불가

**coding** (개발용):
- messaging 전체 +
- `fs_read`, `fs_write`, `fs_list`, `fs_search`
- `exec_run` (exec-approvals 적용)
- `git_status`, `git_diff`, `git_commit`

**full** (모든 권한):
- coding 전체 +
- `agent_send` (에이전트 간 메시지)
- `sessions_spawn`, `sessions_list`
- `cron_create`, `cron_delete`
- `elevated_*` (관리자 도구)

### deny로 세부 차단

프로파일에 포함되어 있어도 `deny`로 특정 도구를 차단할 수 있습니다.

```json5
tools: {
  profile: "full",
  deny: [
    "group:automation",    // 그룹 단위 차단 (cron_*, sessions_*)
    "sessions_spawn",      // 개별 도구 차단
    "elevated_shutdown"    // 위험한 관리 도구 차단
  ]
}
```

### fs.workspaceOnly

```json5
fs: { workspaceOnly: true }
```

`true`(기본)이면 에이전트는 **자기 워크스페이스 디렉토리 안에서만** 파일을 읽고 쓸 수 있습니다. 워크스페이스 밖의 파일에 접근하려 하면 오류가 발생합니다.

`false`로 설정하면 시스템 전체 파일에 접근 가능하지만, **프로덕션에서는 권장하지 않습니다**.
