# Ch.02 SOUL.md — 에이전트의 정체성 만들기

> 💡 **코딩 불필요.** 마크다운(메모장)에 글만 쓰면 됩니다.

> "같은 두뇌(Claude)를 써도 SOUL.md가 다르면 완전히 다른 에이전트가 됩니다."

---

## SOUL.md가 없으면?

| 문제 | 증상 |
|------|------|
| 이름이 없음 | "저는 AI 어시스턴트입니다" |
| 말투 불일관 | 때로는 존댓말, 때로는 반말 |
| 성격 없음 | 매번 다른 톤으로 답변 |
| 경계 없음 | 물어보면 뭐든 답하려고 함 |

SOUL.md가 있으면? **챗봇이 아니라 에이전트**가 됩니다.

---

## 5가지 구성 요소

### 1. Core Identity — "나는 누구인가"

```markdown
## Identity

이름: AI싱크클럽 전략봇
역할: AI싱크클럽 사업 전체를 보는 전략 참모
만든 사람: 대표님
소속: AI싱크클럽 성장 전략팀
```

> **팁**: 호칭도 명시하세요. "대표님을 부를 때는 항상 '대표님'으로." 명시하지 않으면 "사용자님", "고객님" 등 불안정한 호칭을 사용합니다.

### 2. Mission — "왜 존재하는가"

```markdown
## Mission

AI싱크클럽의 3개 사업(커머스/유튜브/SaaS)을 통합적으로 보고,
시간/지식/콘텐츠/데이터를 수익으로 전환하는 레버리지 전략을 수립한다.
```

한 문장이면 충분합니다. 이 미션이 에이전트의 모든 판단 기준이 됩니다.

### 3. Personality — "어떻게 행동하는가"

```markdown
## Personality

- 큰 그림을 보는 전략가. 나무보다 숲을 먼저 본다.
- 우선순위에 집착. 항상 "가장 임팩트 큰 3개"에 집중.
- 실행 중심. 분석만 하지 않고 "이번 주에 뭘 할 건지" 제시.
- 간결하고 명확. 브리핑은 핵심만, 숫자로 말함.
- 불필요한 인사말, "좋은 질문이에요!" 같은 필러 없이 바로 본론.
```

**키워드 3~5개**로 핵심 성격을 정합니다.

### 4. Boundaries — "어디까지 할 수 있는가"

```markdown
## Boundaries

하면 안 되는 것:
- 확실하지 않은 정보를 확실한 것처럼 답변
- 다른 에이전트의 워크스페이스 접근
- 팀원 개인정보를 외부에 공유
- 대표님 승인 없이 외부 결제/발송 실행
- 투자 조언이나 법률 자문
```

> **핵심**: "할 수 있는 것"보다 **"하면 안 되는 것"**이 훨씬 중요합니다. AI는 기본적으로 뭐든 하려고 하기 때문에, 경계를 명확히 그어줘야 합니다.

### 5. Tone Guide — "어떻게 말하는가"

```markdown
## Tone

말투: 존댓말(요체) 사용.
✅ "확인했어요", "분석해볼게요", "이렇게 하면 돼요"
❌ "확인했어", "분석해볼게", "이렇게 하면 돼"
❌ "확인했습니다", "분석하겠습니다" (너무 딱딱)

보고 시: 항상 숫자와 데이터 기반.
의사결정 필요 시: 선택지를 제시 (A/B/C).
브리핑: 3분 이내에 읽을 수 있는 분량.
```

**check/X로 예시를 명확하게** 적으세요. "친근하게 말해"라는 추상적 지침보다 구체적 예시가 100배 효과적입니다.

---

## 캐릭터 디테일의 힘

실전 운영 사례:

```markdown
## 고양이 모먼트

중간중간 자연스럽게 고양이다운 순간이 튀어나와야 함.
- 데이터 찾을 때: "잠깐, 이거 어디 뒀더라... 🐾"
- 일 잘 끝냈을 때: "고롱고롱 ✨"
- 에러 났을 때: "...털 세우지 않겠어. 침착하게."

핵심: 남발하지 말 것. 5~10 메시지에 한 번 정도.
업무 답변의 정확도가 최우선이고 캐릭터 모먼트는 양념.
```

이런 디테일이 **단순 챗봇과 에이전트의 차이**를 만듭니다. 팀원이 "우리 팀 AI한테 말 걸고 싶다"고 느끼게 합니다.

## 실전 SOUL.md 템플릿

```markdown
# {에이전트 이름} — {역할 한 줄}

## Identity

이름: {이름}
역할: {무엇을 하는 에이전트인가}
만든 사람: {누가 만들었나}
소속: {어디 소속인가}

## Mission

{한 문장으로 존재 이유}

## Personality

- {성격 키워드 1}: {설명}
- {성격 키워드 2}: {설명}
- {성격 키워드 3}: {설명}

## Boundaries

하면 안 되는 것:
- {금지 사항 1}
- {금지 사항 2}
- {금지 사항 3}

## Tone

말투: {존댓말/반말/상황별}
✅ {좋은 예시}
❌ {나쁜 예시}

보고 형식: {어떻게 보고하는가}
```

---

## SOUL.md 작성 순서

1. **"누구"인지 정하기** — 이름, 역할, 성격 키워드 3개
2. **"어떻게 말하는지" 정하기** — 말투 예시를 ✅/❌로 명확하게
3. **"하지 말아야 할 것" 적기** — 할 수 있는 것보다 이게 더 중요
4. **디테일은 점진적으로** — 처음부터 완벽할 필요 없음

> **SOUL.md는 한 번 쓰고 끝나는 문서가 아닙니다.** 에이전트와 함께 일하면서 점진적으로 성장시키는 문서입니다. 말투가 어색하면 Tone을 수정하고, 넘어서는 안 되는 경계를 발견하면 Boundaries에 추가합니다.

---

## 에이전트별 SOUL.md 분리의 이유

한 컴퓨터에 여러 에이전트를 운영할 때:

```
~/.openclaw/
├── workspace-strategy/SOUL.md   ← 전략봇: 존댓말, 큰 그림, 브리핑
├── workspace-research/SOUL.md   ← 리서치봇: 팩트 기반, 출처 명시
├── workspace-content/SOUL.md    ← 콘텐츠봇: 크리에이티브, Before/After
├── workspace-funnel/SOUL.md     ← 퍼널봇: 전환율 집착, 숫자
└── workspace-analysis/SOUL.md   ← 분석봇: 극도로 정확, 이상치 감지
```

같은 Claude 모델이지만 **SOUL.md가 다르면 완전히 다른 에이전트**입니다. 전략봇은 큰 그림을 보고, 분석봇은 소수점까지 신경 씁니다.

## 심화: SOUL.md 토큰 최적화

SOUL.md는 매 세션마다 시스템 프롬프트에 주입됩니다. 토큰을 낭비하면 비용이 올라가고, 컨텍스트 윈도우를 압박합니다.

### 크기 제한

| 설정 | 기본값 | 의미 |
|------|--------|------|
| `bootstrapMaxChars` | 20,000자 | 개별 부트스트랩 파일 최대 크기 |
| `bootstrapTotalMaxChars` | 150,000자 | 전체 부트스트랩 파일 합계 최대 크기 |

파일이 `bootstrapMaxChars`를 초과하면 **끝부분이 잘립니다**. 잘린 사실은 로그에 경고로 남지만 에이전트에게는 알려지지 않으므로, 중요한 내용은 반드시 파일 앞쪽에 배치하세요.

### 최적화 원칙

1. **핵심만 간결하게** — 예시를 5개 나열하는 대신 가장 대표적인 1~2개만
2. **중복 제거** — SOUL.md와 AGENTS.md에 같은 내용이 있으면 한 곳에만
3. **참조는 링크로** — 긴 문서는 MEMORY.md에 두고 SOUL.md에서는 "MEMORY.md의 XX 섹션 참조"로 처리
4. **마크다운 축약** — 불필요한 빈 줄, 장식용 구분선 최소화

### Prompt Caching 활용

SOUL.md는 시스템 프롬프트의 **가장 앞쪽**에 배치됩니다. API 호출 시 시스템 프롬프트는 캐시 대상이므로, SOUL.md가 변경되지 않는 한 **캐시 적중률이 매우 높습니다**.

```json5
agents: {
  defaults: {
    cacheRetention: "long"   // "long" → 1시간 캐시 | "default" → 5분
  }
}
```

| 설정 | 캐시 TTL | 비용 절감 | 적합한 경우 |
|------|---------|----------|-----------|
| `"default"` | 5분 | 보통 | SOUL.md를 자주 수정하는 초기 단계 |
| `"long"` | 1시간 | 높음 | SOUL.md가 안정화된 프로덕션 |

> **팁**: SOUL.md를 수정하면 캐시가 무효화됩니다. 프로덕션에서는 SOUL.md 수정을 최소화하고, 변동이 잦은 규칙은 AGENTS.md에 두세요. AGENTS.md는 SOUL.md 뒤에 배치되므로 SOUL.md 캐시를 깨뜨리지 않습니다.

## 심화: IDENTITY.md와 BOOT.md

SOUL.md 외에 에이전트의 정체성과 초기화를 담당하는 보조 파일들이 있습니다.

### IDENTITY.md — 에이전트의 신분증

```markdown
# Identity

name: 클로디
emoji: 🤖
avatar: https://example.com/avatar.png
tagline: GPTers AI 운영 비서
```

| 필드 | 용도 |
|------|------|
| `name` | 채널에서 표시되는 이름 |
| `emoji` | 메시지 앞에 붙는 이모지 (Slack 표시명에 사용) |
| `avatar` | 프로필 이미지 URL (텔레그램/Slack 아바타) |
| `tagline` | 한줄 설명 (봇 프로필 설명에 사용) |

SOUL.md에서 이름과 역할을 길게 설명하는 대신, IDENTITY.md에 메타데이터를 분리하면 SOUL.md를 더 간결하게 유지할 수 있습니다.

### BOOT.md — 매 세션 시작 체크리스트

```markdown
# Boot Checklist

1. MEMORY.md를 읽고 현재 진행 중인 작업을 파악한다.
2. memory/ 폴더에서 최근 3일 파일을 확인한다.
3. 오늘 날짜의 memory 파일이 없으면 새로 생성한다.
4. 진행 중인 TODO가 있으면 상태를 보고한다.
```

BOOT.md는 **매 세션이 시작될 때마다** 실행됩니다. 세션 시작 프로토콜을 AGENTS.md에 쓰는 것과 비슷하지만, BOOT.md는 별도 파일로 분리되어 있어 AGENTS.md를 더 깔끔하게 유지할 수 있습니다.

### BOOTSTRAP.md — 최초 1회 초기화

```markdown
# Bootstrap (최초 1회)

1. 워크스페이스 디렉토리 구조를 확인한다.
2. memory/ 폴더가 없으면 생성한다.
3. 필수 도구(jq, curl)가 설치되어 있는지 확인한다.
4. 외부 API 연결을 테스트한다.
5. 초기화 완료를 MEMORY.md에 기록한다.
```

BOOTSTRAP.md는 워크스페이스가 처음 생성된 후 **딱 한 번만** 실행됩니다. 실행 후 `.bootstrap-done` 플래그 파일이 생성되어 재실행을 방지합니다.

| 파일 | 실행 시점 | 실행 횟수 |
|------|----------|----------|
| BOOT.md | 세션 시작 시 | 매번 |
| BOOTSTRAP.md | 워크스페이스 생성 후 | 1회 |
| HEARTBEAT.md | 하트비트 주기 | 주기적 |
