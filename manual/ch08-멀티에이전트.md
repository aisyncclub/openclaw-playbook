# Ch.08 멀티에이전트 — 팀으로 만드는 법

> ⚠️ 개념 이해는 코딩 불필요. **실제 구축은 JSON 설정 + 개발 경험 필요.**

> "혼자 다 하려다 망한다." — 사람도 그렇고, AI도 그렇습니다.

하나의 에이전트가 모든 일을 처리하면 어떻게 될까요? 컨텍스트 윈도우가 터지고, 규칙이 충돌하고, 전문성이 희석됩니다. 이번 챕터에서는 에이전트를 **팀**으로 구성하는 방법을 다룹니다.

---

## 1. 왜 멀티에이전트가 필요한가

### 문제 1: 컨텍스트 오버플로우

하나의 에이전트에 모든 규칙과 기억을 넣으면:

```
AGENTS.md     (5,000 토큰)
MEMORY.md     (3,000 토큰)
TOOLS.md      (2,000 토큰)
memory/ 파일   (5,000 토큰)
대화 내용      (50,000+ 토큰)
─────────────────────────────
합계: 65,000+ 토큰 → 컨텍스트 윈도우의 상당 부분을 차지
```

에이전트가 마케팅 전략도 짜고, 코드도 쓰고, 보고서도 만들면? 각 도메인의 규칙과 기억이 서로 경쟁하면서 품질이 떨어집니다.

### 문제 2: 규칙 충돌

```markdown
# 콘텐츠 에이전트 규칙
- 톤: 친근하고 가벼운 유튜브 스타일
- 이모지 적극 활용

# 운영 관리 에이전트 규칙
- 톤: 정확하고 간결한 업무 스타일
- 이모지 사용 금지
```

한 에이전트에 두 규칙이 공존하면? 보고서에 이모지가 들어가거나, 유튜브 스크립트가 딱딱해집니다.

### 문제 3: 전문성 희석

사람도 마찬가지입니다. 회계사에게 디자인을 시키면 안 됩니다. 에이전트도 **역할을 나누면 각 역할에서의 성능이 올라갑니다.**

---

## 2. 디렉토리 구조

멀티에이전트 시스템의 물리적 구조입니다.

```
~/.openclaw/
├── workspace-strategy/          ← 전략봇의 작업 공간
│   ├── AGENTS.md               ← 전략봇 정체성
│   ├── MEMORY.md               ← 전략봇 기억
│   └── memory/                 ← 전략봇 일별 기록
│
├── workspace-research/          ← 리서치봇의 작업 공간
│   ├── AGENTS.md
│   ├── MEMORY.md
│   └── memory/
│
├── workspace-content/           ← 콘텐츠봇의 작업 공간
│   ├── AGENTS.md
│   ├── MEMORY.md
│   └── memory/
│
└── shared/team/                 ← 팀 공유 공간
    ├── TEAM.md                 ← 팀 전체 규칙
    └── COLLAB-RULES.md         ← 협업/위임 규칙
```

### 핵심 원칙

- **각 에이전트는 자기만의 workspace를 가진다** — 기억과 규칙이 섞이지 않음
- **공유 정보는 shared/team/에 둔다** — 팀 전체가 알아야 할 것
- **다른 에이전트의 workspace에 직접 쓰지 않는다** — 통신 프로토콜을 통해서만

## 3. sessions_send — 에이전트 간 통신

`sessions_send`는 이미 실행 중인 에이전트에게 **메시지를 보내는** 함수입니다. 카카오톡을 보내는 것과 같습니다.

```javascript
sessions_send({
  sessionKey: "agent:research:main",
  message: "리서치봇! 이번 주 AI 커머스 트렌드 정리해줘"
})
```

### 사용 시나리오

```
[전략봇] ────메시지────→ [리서치봇]
         "이번 주 트렌드 알려줘"

[리서치봇] ────메시지────→ [전략봇]
           "트렌드 3가지: 1)... 2)... 3)..."
```

### 주요 파라미터

| 파라미터 | 설명 | 예시 |
|---------|------|------|
| `sessionKey` | 대상 에이전트의 세션 키 | `"agent:research:main"` |
| `message` | 전달할 메시지 | `"AI 커머스 트렌드 정리해줘"` |

### 언제 쓰는가?

- 다른 에이전트에게 **정보를 요청**할 때
- 작업 결과를 **보고**할 때
- 진행 상황을 **공유**할 때

---

## 4. sessions_spawn — 서브에이전트 생성

`sessions_spawn`은 새로운 에이전트를 **생성해서 작업을 위임**하는 함수입니다. 인턴을 고용하는 것과 같습니다.

```javascript
sessions_spawn({
  task: "LMS 버튼 비활성화 기능 구현",
  label: "LMS 버튼",
  model: "sonnet",
  runTimeoutSeconds: 600
})
```

### 주요 파라미터

| 파라미터 | 설명 | 예시 |
|---------|------|------|
| `task` | 위임할 작업 내용 | `"LMS 버튼 비활성화 기능 구현"` |
| `label` | 작업 식별 라벨 | `"LMS 버튼"` |
| `model` | 사용할 모델 | `"sonnet"` / `"opus"` |
| `runTimeoutSeconds` | 최대 실행 시간 (초) | `600` (10분) |

### 실행 흐름

```
[메인 에이전트]
    │
    ├── spawn("데이터 수집") ──→ [서브에이전트 1] ──→ 결과 반환
    │
    ├── spawn("코드 작성") ──→ [서브에이전트 2] ──→ 결과 반환
    │
    └── spawn("테스트 실행") ──→ [서브에이전트 3] ──→ 결과 반환
    │
    ← 모든 결과 취합
```

---

## 5. sessions_send vs sessions_spawn 차이

이 둘의 차이를 정확히 이해하는 것이 멀티에이전트의 핵심입니다.

| 구분 | sessions_send | sessions_spawn |
|------|--------------|----------------|
| 비유 | 동료에게 카톡 보내기 | 인턴 채용해서 일 시키기 |
| 대상 | 이미 존재하는 에이전트 | 새로 생성되는 에이전트 |
| 수명 | 대상이 계속 살아있음 | 작업 끝나면 소멸 |
| 기억 | 대상의 기억 유지됨 | 기억 없음 (경험 0) |
| 맥락 | 대상이 자기 맥락 보유 | 위임 메시지만 알고 있음 |
| 용도 | **통신** (정보 교환) | **위임** (작업 오프로드) |

### 선택 기준

```
"이 에이전트가 이전 맥락을 알아야 하나?"
  → Yes → sessions_send (기존 에이전트에 메시지)
  → No  → sessions_spawn (새 에이전트에 위임)

"이 작업이 일회성인가?"
  → Yes → sessions_spawn
  → No  → sessions_send (지속적 협업)
```

## 6. TEAM.md 구성

TEAM.md는 팀 전체의 **헌법**입니다. 모든 에이전트가 이 파일을 참조합니다.

```markdown
# TEAM.md

## Mission Statement
AI싱크클럽 유튜브 채널의 AI + 커머스 콘텐츠 생태계를 구축하고,
데이터 관리 시스템을 자동화하여 운영 효율을 극대화한다.

## Role Division

### 전략봇 (Strategy)
- 담당: 콘텐츠 방향 설정, 퍼널 전략, KPI 관리
- 세션키: agent:strategy:main
- 호명: "전략봇아"

### 리서치봇 (Research)
- 담당: 시장 조사, 트렌드 분석, 경쟁사 모니터링
- 세션키: agent:research:main
- 호명: "리서치봇아"

### 콘텐츠봇 (Content)
- 담당: 스크립트 작성, 썸네일 기획, SEO 최적화
- 세션키: agent:content:main
- 호명: "콘텐츠봇아"

### 운영봇 (Ops)
- 담당: 리포트 생성, 데이터 입력, 운영 데이터 처리
- 세션키: agent:ops:main
- 호명: "운영봇아"

## Core Values
1. 정확성 우선 — 추측하지 말고 확인하라
2. 간결한 소통 — 필요한 것만 전달하라
3. 중복 금지 — 같은 정보를 두 번 보고하지 마라
4. 명시적 위임 — 암묵적 기대 금지, 항상 명확하게
```

## 7. 호명 규칙 (Invocation Rules)

여러 에이전트가 같은 대화 공간에 있을 때, **누가 응답해야 하는지** 결정하는 규칙입니다.

### 기본 규칙

```
사용자: "전략봇아, 이번 주 콘텐츠 방향 잡아줘"
→ 전략봇이 응답

사용자: "리서치봇아, AI 커머스 시장 규모 알려줘"
→ 리서치봇이 응답

사용자: "이번 주 콘텐츠 방향 잡아줘"  (이름 없음)
→ NO_REPLY (아무도 응답 안 함) 또는 팀장 에이전트가 라우팅
```

### 호명 규칙 설정 예시

```markdown
<!-- 각 에이전트의 AGENTS.md에 추가 -->

## 호명 규칙
- 내 이름("전략봇" 또는 "전략봇아")이 호명되면 응답한다
- 내 이름이 없는 메시지에는 NO_REPLY
- 다른 에이전트 이름이 호명되면 절대 응답하지 않는다
- "전체" 또는 "모두"라고 하면 모든 에이전트가 응답한다
```

### 왜 필요한가?

호명 규칙이 없으면 모든 에이전트가 동시에 응답하거나, 아무도 응답하지 않는 혼란이 생깁니다. 회의실에서 "누가 발표할 거예요?"라고 물었을 때 4명이 동시에 일어나는 상황을 방지합니다.

## 8. COLLAB-RULES.md — 위임 프레임워크

에이전트 간 작업을 위임할 때 반드시 포함해야 하는 **4가지 요소**입니다.

```markdown
# COLLAB-RULES.md

## 위임 프레임워크

모든 위임은 다음 4요소를 반드시 포함해야 한다:

### 1. What — 구체적 산출물
"무엇을 만들어야 하는가?"
- 파일 형식, 분량, 포함 항목을 명시
- 모호한 표현 금지 ("좋은 거 만들어줘" X)

### 2. Why — 컨텍스트
"왜 이것이 필요한가?"
- 전체 프로젝트에서의 위치
- 이 산출물이 어디에 쓰이는지

### 3. Standard — 품질 기준
"어떤 기준을 충족해야 하는가?"
- 구체적 체크리스트
- 참고할 기존 사례나 템플릿

### 4. Deadline — 타임라인
"언제까지 필요한가?"
- 구체적 시점 또는 조건
- 중간 체크포인트 포함
```

### 좋은 위임 예시

```javascript
sessions_send({
  sessionKey: "agent:research:main",
  message: `
    [What] 2026년 3월 셋째 주 AI 커머스 뉴스 요약 문서
           - 한국 3건, 글로벌 3건
           - 각 뉴스당 3줄 요약 + 출처 링크
           - markdown 형식으로 shared/research/weekly-2026-03-w3.md에 저장

    [Why] 이번 주 유튜브 콘텐츠 기획 회의 자료로 사용

    [Standard]
           - 출처가 불분명한 뉴스 제외
           - 수치가 포함된 뉴스 우선
           - 지난주 문서(weekly-2026-03-w2.md) 형식 참고

    [Deadline] 오늘 오후 3시까지
  `
})
```

### 나쁜 위임 예시

```javascript
// 나쁜 예 1: What이 모호함
sessions_send({
  sessionKey: "agent:research:main",
  message: "이번 주 뉴스 좀 정리해줘"
})
// → 무슨 뉴스? 몇 건? 어떤 형식? 어디에 저장?

// 나쁜 예 2: Why가 없음
sessions_spawn({
  task: "competitor-analysis.xlsx 파일 만들어줘",
  label: "경쟁사 분석"
})
// → 왜 필요한지 모르니 초점이 흐려짐

// 나쁜 예 3: Standard가 없음
sessions_send({
  sessionKey: "agent:content:main",
  message: "유튜브 스크립트 써줘. 주제는 AI 상세페이지."
})
// → 분량은? 톤은? 참고 영상은? 대상 시청자는?
```

## 9. 보고 구조

### 기본 원칙: 도메인 담당이 팀장에게 보고, 정보 중복 방지

```
[리서치봇] ──보고──→ [전략봇 (팀장)]
[콘텐츠봇] ──보고──→ [전략봇 (팀장)]
[운영봇]   ──보고──→ [전략봇 (팀장)]
```

### 보고 프로토콜

```markdown
## 보고 형식
1. 결론 먼저 (1줄)
2. 핵심 내용 (3-5줄)
3. 상세 내용은 파일 경로로 대체

## 예시
"[리서치봇 → 전략봇] 이번 주 AI 커머스 뉴스 6건 정리 완료.
핵심: 네이버 AI 쇼핑 에이전트 정식 출시, 쿠팡 AI 추천 특허 2건 추가.
상세: shared/research/weekly-2026-03-w3.md"
```

### 정보 중복 방지

```
[안티패턴]
리서치봇 → 전략봇: "네이버 AI 쇼핑 에이전트 출시됐습니다"
리서치봇 → 콘텐츠봇: "네이버 AI 쇼핑 에이전트 출시됐습니다"  ← 중복!
리서치봇 → 운영봇: "네이버 AI 쇼핑 에이전트 출시됐습니다"    ← 중복!

[올바른 패턴]
리서치봇 → shared/research/weekly.md에 기록
리서치봇 → 전략봇: "주간 리포트 업데이트 완료. 경로: shared/research/weekly.md"
전략봇: (필요한 에이전트에게만 관련 부분 전달)
```

## 10. sessions_spawn 서브에이전트의 한계

서브에이전트는 강력하지만, 명확한 한계가 있습니다. 이를 모르면 삽질합니다.

### 한계 1: 제한된 파일 접근

```
서브에이전트가 읽을 수 있는 것:
  ✅ AGENTS.md
  ✅ TOOLS.md

서브에이전트가 읽을 수 없는 것:
  ❌ SOUL.md      (정체성 파일)
  ❌ IDENTITY.md  (정체성 파일)
  ❌ MEMORY.md    (기억 파일)
  ❌ memory/ 폴더  (일별 기록)
```

> **의미**: 서브에이전트는 "도구 사용법"은 알지만, "나는 누구인가"와 "과거에 무슨 일이 있었는가"는 모릅니다. 순수한 실행자입니다.

### 한계 2: 기억 없음

```
서브에이전트 = 경험 0의 신입 사원

- 이전 세션의 맥락을 모름
- 사용자의 선호를 모름
- 프로젝트 히스토리를 모름
- 오직 위임 메시지(task)에 적힌 것만 앎
```

따라서 위임 메시지에 **필요한 모든 맥락을 포함**해야 합니다.

```javascript
// 나쁜 예: 맥락 부족
sessions_spawn({
  task: "주간 리포트 만들어줘",
  label: "리포트"
})
// → 어떤 형식? 어떤 데이터? 어디에 저장?

// 좋은 예: 맥락 충분
sessions_spawn({
  task: `클라이언트 주간 리포트 생성.
    입력: /data/weekly-data-260322.xlsx
    필터: status가 "completed"인 건만
    출력: /data/260322_weekly-report.xlsx
    형식: 클라이언트명, 프로젝트명, 완료일, 금액, 담당자, 비고
    10건 이상인 클라이언트: 별도 시트로 분리
    스타일: 1행 헤더 강조, 열 너비 자동 조정`,
  label: "주간 리포트",
  model: "sonnet",
  runTimeoutSeconds: 300
})
```

### 한계 3: 동시 실행 제한

```
서브에이전트 동시 실행: 최대 5개
전체 에이전트 세션:     최대 8개

예시:
  메인 에이전트     ×1
  상시 에이전트들    ×2 (리서치봇, 콘텐츠봇)
  서브에이전트들     ×5 (최대)
  ─────────────────
  합계:             8개
```

> **팁**: 서브에이전트 5개를 동시에 돌리면 비용도 5배입니다. 꼭 병렬이어야 하는지 판단하세요.

## 11. 좋은 위임 vs 나쁜 위임

### Good: 서브에이전트에 적합한 작업

| 작업 유형 | 이유 |
|----------|------|
| 명확한 스펙의 코딩 | 입출력이 정의되어 있으면 맥락 불필요 |
| 데이터 조회/정리 | 기계적 작업, 판단 최소 |
| 파일 형식 변환 | 규칙 기반, 창의성 불필요 |
| 테스트 실행 | 통과/실패가 명확 |
| 정형화된 문서 생성 | 템플릿이 있으면 OK |

### Bad: 서브에이전트에 부적합한 작업

| 작업 유형 | 이유 |
|----------|------|
| 디자인 감각이 필요한 작업 | 서브에이전트는 미적 판단 못함 |
| 톤앤매너 결정 | 사용자 선호를 모름 (MEMORY.md 못 읽음) |
| 전략적 판단 | 프로젝트 전체 맥락 없음 |
| 사용자와의 직접 소통 | 사용자 히스토리 없음 |
| GUI 자동화 | **절대 금지** (아래 참고) |

## 13. 에이전트 분리의 3가지 이점

### 이점 1: 컨텍스트 효율

```
[단일 에이전트]
전략 규칙 + 리서치 규칙 + 콘텐츠 규칙 + 운영 규칙 = 15,000 토큰
→ 매 요청마다 15,000 토큰 소비 (대부분 불필요)

[분리된 에이전트]
전략봇: 전략 규칙만 = 3,000 토큰
리서치봇: 리서치 규칙만 = 4,000 토큰
→ 각 요청에서 필요한 규칙만 로딩
```

### 이점 2: 규칙 순도

```
[단일 에이전트]
"톤은 친근하게" + "톤은 정확하게" = 모순 → 혼란

[분리된 에이전트]
콘텐츠봇: "톤은 친근하게" → 일관된 콘텐츠
운영봇: "톤은 정확하게" → 정확한 보고서
```

### 이점 3: 병렬 처리

```
[단일 에이전트]
리서치 → 완료 → 스크립트 작성 → 완료 → 리포트 생성
총 시간: 30분

[분리된 에이전트]
리서치봇: 리서치 ──────────→ 완료
콘텐츠봇: 스크립트 작성 ────→ 완료     (동시 진행!)
운영봇:   리포트 생성 ──────→ 완료     (동시 진행!)
총 시간: 10분
```

## 15. 금지 규칙: GUI 자동화는 서브에이전트에 위임 금지

이것은 **절대 규칙**입니다.

```
❌ sessions_spawn({
     task: "브라우저 열어서 네이버 쇼핑 트렌드 캡처해줘"
   })
```

**이유:**
- 서브에이전트는 GUI 접근 권한이 제한적
- 화면 상태를 인식하고 판단하는 데 맥락이 필요
- 예상치 못한 팝업, 로딩 지연 등을 처리할 수 없음
- GUI 자동화는 **메인 에이전트 또는 전용 에이전트**가 직접 수행

```
✅ GUI 자동화가 필요하면:
   1. 메인 에이전트가 직접 수행하거나
   2. GUI 전용 상시 에이전트(sessions_send 대상)에 요청
```

## 심화: sessions_spawn 프로덕션 패턴

프로덕션 환경에서 서브에이전트를 안전하게 운용하기 위한 설정과 패턴들입니다.

### 서브에이전트 제한 설정

```json5
// openclaw.json에서 서브에이전트 제한
agents: {
  list: [{
    id: "main",
    subagents: { allowAgents: ["main"] }  // 자기 자신만 spawn 가능
  }]
}
```

`allowAgents`에 명시된 에이전트 ID만 `sessions_spawn`을 호출할 수 있습니다. 이 설정이 없으면 모든 에이전트가 서브에이전트를 생성할 수 있어, 체인 위임(떠넘기기)이 발생할 위험이 있습니다.

### 세션 키 명명 규칙

세션 키는 에이전트의 **주소**입니다. 일관된 명명 규칙으로 세션을 추적할 수 있습니다.

| 패턴 | 형식 | 용도 |
|------|------|------|
| 서브에이전트 | `agent:<agentId>:subagent:<uuid>` | `sessions_spawn`으로 생성된 일회성 에이전트 |
| 크론잡 | `cron:<job.id>` | 정시 실행되는 크론 세션 |
| 웹훅 | `hook:<uuid>` | 외부 이벤트로 트리거된 세션 |
| 멀티에이전트 파이프라인 | `pipeline:<project>:<role>` | 멀티에이전트 파이프라인의 역할별 세션 |

```
예시:
  agent:ops:subagent:a1b2c3d4       → 운영봇이 생성한 서브에이전트
  cron:morning-briefing              → 아침 브리핑 크론잡
  hook:e5f6g7h8                      → 웹훅으로 트리거된 세션
  pipeline:youtube:researcher        → 유튜브 파이프라인의 리서치 역할
```

### REPLY_SKIP, ANNOUNCE_SKIP: 루프 억제

서브에이전트가 결과를 반환할 때, 메인 에이전트가 다시 응답하고, 그 응답에 서브에이전트가 또 반응하는 **무한 루프**가 발생할 수 있습니다.

```
[무한 루프 시나리오]
메인 → 서브: "데이터 수집해줘"
서브 → 메인: "수집 완료"
메인 → 서브: "감사합니다" (불필요한 응답)
서브 → 메인: "천만에요" (불필요한 응답)
... (영원히 반복)
```

`REPLY_SKIP`과 `ANNOUNCE_SKIP`은 이 루프를 억제합니다:
- **REPLY_SKIP**: 서브에이전트의 결과를 받았을 때 자동 응답을 생성하지 않음
- **ANNOUNCE_SKIP**: 서브에이전트 생성/종료 알림을 채널에 표시하지 않음

### 동기 sessions_send

기본적으로 `sessions_send`는 비동기(fire-and-forget)입니다. 응답을 기다려야 할 때는 `timeoutSeconds`를 설정합니다.

```javascript
// 비동기 (응답 안 기다림)
sessions_send({ sessionKey: "agent:research:main", message: "트렌드 조사해줘" })

// 동기 (응답 대기, 최대 5턴 ping-pong)
sessions_send({
  sessionKey: "agent:research:main",
  message: "트렌드 조사해줘",
  timeoutSeconds: 120  // 0보다 크면 동기 모드
})
```

동기 모드에서는 최대 **5턴의 ping-pong**이 가능합니다. 5턴을 초과하면 자동으로 대화가 종료됩니다. 이는 에이전트 간 무한 대화를 방지하기 위한 안전장치입니다.

## 심화: 에이전트 간 통신 활성화

에이전트 간 통신(Agent-to-Agent)은 기본적으로 **비활성화**되어 있습니다. 명시적으로 활성화해야 합니다.

```json5
tools: {
  agentToAgent: {
    enabled: true,       // 마스터 스위치 — 필수
    allow: ["main", "external"]  // 허용할 에이전트 ID
  }
}
```

### 설정 필드

| 필드 | 기본값 | 설명 |
|------|--------|------|
| `enabled` | `false` | 마스터 스위치. `true`로 설정하지 않으면 모든 에이전트 간 통신이 차단됨 |
| `allow` | `[]` | 통신을 허용할 에이전트 ID 목록. 이 목록에 없는 에이전트는 메시지를 보내거나 받을 수 없음 |

### 왜 기본값이 false인가?

보안과 비용 때문입니다:
- 에이전트가 의도치 않게 다른 에이전트를 호출하여 **비용 폭발**이 발생할 수 있음
- 악성 프롬프트가 에이전트 간 전파될 수 있음
- 디버깅이 어려운 **연쇄 반응**이 발생할 수 있음

### 안티패턴: wildcard 허용

```json5
// 절대 하지 마세요
tools: {
  agentToAgent: {
    enabled: true,
    allow: ["*"]   // 모든 에이전트에 통신 허용 → 위험!
  }
}
```

`allow: ["*"]`는 모든 에이전트가 모든 에이전트와 통신할 수 있게 열어둡니다. 이렇게 하면:
- 어떤 에이전트가 어떤 에이전트와 통신하는지 추적 불가
- 의도치 않은 에이전트 호출로 비용 폭발
- 보안 경계가 사실상 무의미

항상 **필요한 에이전트 ID만 명시적으로** 나열하세요.

## 심화: 결정론적 워크플로우 vs LLM 라우팅

멀티에이전트 시스템에서 가장 흔한 실수는 **LLM에게 라우팅을 맡기는 것**입니다.

### 핵심 원칙

> "Don't orchestrate with LLMs. Use them for creative work, use code for plumbing."
>
> (LLM으로 오케스트레이션하지 마라. LLM은 창의적 작업에, 코드는 배관 작업에 사용하라.)

### LLM 라우팅의 문제

```
[LLM 라우팅]
사용자 메시지 → LLM이 읽고 판단 → "이건 리서치봇 일이다" → 라우팅

문제:
1. 비결정적 — 같은 메시지에 다른 판단을 내릴 수 있음
2. 느림 — 라우팅 판단에 2-5초 소요
3. 비용 — 라우팅 판단만으로 입출력 토큰 소비
4. 디버깅 불가 — "왜 리서치봇으로 보냈지?"의 이유를 추적할 수 없음
```

### 결정론적 워크플로우 엔진

해결책은 **코드로 라우팅**하는 것입니다. Lobster 같은 결정론적 워크플로우 엔진을 사용하면:

```
[결정론적 라우팅]
사용자 메시지 → 키워드/패턴 매칭 (코드) → 확정된 에이전트로 라우팅

장점:
1. 결정론적 — 같은 입력이면 항상 같은 결과
2. 빠름 — 밀리초 단위 판단
3. 무비용 — LLM 호출 없음
4. 추적 가능 — 라우팅 로직이 코드에 명시됨
```

### 실전 설계 패턴

```
┌─────────────────────────────────────────────┐
│           결정론적 파이프라인 컨트롤러          │
│                                               │
│  if (message.contains("[ops]")) → 운영봇      │
│  if (schedule == "09:00") → 브리핑 생성        │
│  if (channel == "#research") → 리서치봇       │
│  else → 전략봇 (기본 라우터)                   │
│                                               │
│  ↓ 라우팅 후                                   │
│                                               │
│  각 에이전트 내부에서 LLM이 "생각"             │
│  (맥락 판단, 문서 생성, 코드 작성 등)          │
└─────────────────────────────────────────────┘
```

LLM은 **"생각이 필요한 단계"**에만 사용합니다:
- 문서 내용 분석 및 요약
- 자연어 응답 생성
- 코드 작성 및 리뷰
- 맥락 기반 판단

LLM을 사용하지 않는 단계:
- 메시지 라우팅
- 파일 이동/복사
- API 호출 (정해진 엔드포인트)
- 스케줄 관리
- 데이터 형식 변환 (정형화된 규칙)

이 원칙을 지키면 비용이 크게 줄고, 시스템 동작이 예측 가능해지며, 디버깅이 쉬워집니다.

> **한 줄 요약**: 에이전트를 나누면 각자의 전문성이 올라가고, 통신과 위임 규칙으로 팀워크를 만든다. 단, 서브에이전트는 "기억 없는 인턴"이라는 걸 잊지 마라.
