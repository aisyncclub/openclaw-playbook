# Ch.06 스킬(Skill) — 에이전트에게 능력 부여하기

> ⚠️ SKILL.md 작성은 코딩 불필요. **scripts/ 폴더는 코딩 필요.** ClawHub에서 기성 스킬 설치는 코딩 불필요.

> **"n8n 워크플로우 100개를 만들었다. 그리고 전부 버렸다."**

에이전트가 아무리 똑똑해도, 구체적인 업무 수행 방법을 모르면 쓸모가 없습니다. "아침 브리핑 보내줘"라고 했을 때, 어디서 데이터를 가져오고, 어떤 형식으로 정리하고, 어디로 보내야 하는지 — 이 모든 것을 알려주는 것이 **스킬(Skill)**입니다.

---

## 1. 스킬이란 무엇인가

스킬은 에이전트의 **능력 단위**입니다.

사람으로 치면:
- "엑셀 다룰 줄 알아?" → 스킬
- "회의록 작성할 수 있어?" → 스킬
- "고객에게 주간 리포트 보낼 수 있어?" → 스킬

에이전트로 치면:
- "아침 브리핑 생성할 수 있어?" → `daily-briefing` 스킬
- "리더 등록할 수 있어?" → `register-leader` 스킬
- "랜딩페이지 배포할 수 있어?" → `deploy-landing` 스킬

**스킬 = SKILL.md(설명서) + scripts/(실행 코드) + references/(참고 자료)**

---

## 2. n8n 100개 워크플로우를 버린 이유

OpenClaw 스킬 시스템이 탄생한 배경에는 n8n과의 뼈아픈 경험이 있습니다.

### n8n으로 자동화를 구축했을 때 벌어진 일

1. **수정이 너무 어렵다**
   - "브리핑에 매출 항목 하나만 추가해줘" → n8n 에디터 열고, 블록 찾고, JSON 수정하고, 연결 확인하고... 30분
   - 코드였으면? → 3줄 추가, 끝

2. **디버깅이 지옥이다**
   - 12개 블록 중 7번째에서 에러 → 어디서 잘못됐는지 찾으려면 각 블록의 입출력을 하나하나 확인
   - 블록 사이의 데이터 변환 과정이 GUI 뒤에 숨어 있어서 추적이 어려움

3. **문서가 없다**
   - n8n 워크플로우 자체가 문서? 아닙니다. 블록을 하나하나 클릭해봐야 무슨 로직인지 알 수 있음
   - 3개월 뒤에 다시 보면 "이게 왜 이렇게 연결되어 있지?" 상태

4. **맥락적 판단이 불가능하다**
   - n8n은 **정해진 경로**를 따릅니다. A → B → C, 무조건.
   - "평일이면 A, 주말이면 B, 그런데 긴급건이면 C" 같은 복잡한 분기가 늘어날수록 스파게티 워크플로우가 됩니다.

### n8n에서 TypeScript로 변환하면

```
n8n: 12 블록 + 3 분기 + 4 데이터 변환
                    ↓
TypeScript: ~100줄
```

12개 블록이 100줄의 코드가 됩니다. 그리고 그 100줄은:
- `git diff`로 변경 사항을 추적할 수 있고
- 에러가 나면 스택 트레이스로 정확한 위치를 알 수 있고
- 주석과 함수명으로 의도가 명확하게 드러나고
- 에이전트가 **맥락을 보고 판단**할 수 있습니다

## 3. SKILL.md 구조와 작성법

SKILL.md는 에이전트가 읽는 **스킬 설명서**입니다. 에이전트는 이 파일을 읽고 "아, 이 업무는 이렇게 하면 되는구나"를 이해합니다.

### 기본 구조

```markdown
---
name: daily-briefing
description: 매일 아침 3개 사업 현황을 종합하여 브리핑 생성
---

# Daily Briefing 스킬

## 개요
3개 사업(커머스, 교육, 컨설팅)의 현황을 수집하여 아침 브리핑을 생성합니다.

## 실행 조건
- Cron에 의해 매일 09:00 KST에 호출됨
- 수동으로 "브리핑 만들어줘"라고 요청해도 실행 가능

## 실행 순서

### 1단계: 데이터 수집
- Notion "매출 대시보드"에서 어제 매출 데이터 조회
- Notion "일정" 데이터베이스에서 오늘 일정 조회
- Slack #issues 채널에서 미해결 이슈 조회

### 2단계: 브리핑 생성
- 아래 형식으로 마크다운 브리핑 생성:

```
📊 {날짜} 아침 브리핑

| 사업부 | 어제 매출 | 전일 대비 | 주요 이슈 |
|--------|----------|----------|----------|
| 커머스 | {금액} | {증감} | {이슈} |
| 교육 | {금액} | {증감} | {이슈} |
| 컨설팅 | {금액} | {증감} | {이슈} |

📅 오늘 일정
- {일정 목록}

⚠️ 미해결 이슈
- {이슈 목록}
```

### 3단계: 발송
- Slack #morning-briefing 채널에 발송
- 발송 실패 시 → #ops-alert에 에러 보고

## 예외 처리
- Notion API 에러 → "데이터 수집 실패" 메시지와 함께 부분 브리핑 발송
- 매출 데이터가 0원 → 정상 (휴일일 수 있음), 그대로 표시
- Slack 발송 실패 → 3회 재시도 후 로그에 기록

## 참고
- API 키와 채널 ID는 `references/api-guide.md` 참조
- Notion 데이터베이스 ID는 `references/api-guide.md` 참조
```

### SKILL.md 작성 원칙

1. **에이전트가 읽는 문서**입니다. 사람이 아니라 에이전트 관점에서 쓰세요.
2. **실행 순서**를 명확하게 단계별로 적으세요.
3. **예외 처리**를 반드시 적으세요. 에이전트는 예외 상황에서 판단이 필요합니다.
4. **출력 형식**을 구체적으로 적으세요. "적절히 정리해서"가 아니라 템플릿을 제공하세요.

## 4. 스킬 폴더 구조

```
~/.openclaw/skills/{스킬이름}/
├── SKILL.md              ← 에이전트가 읽는 설명서
├── scripts/
│   ├── main.ts           ← 핵심 로직
│   └── helper.ts         ← 보조 기능
├── references/
│   └── api-guide.md      ← API 키, 엔드포인트, ID 매핑
└── .env                  ← 환경변수
```

### 각 파일의 역할

| 파일/폴더 | 역할 | 에이전트가 읽는가? |
|-----------|------|:------------------:|
| `SKILL.md` | 스킬 설명서. 무엇을, 어떻게, 언제 실행하는지 | O (항상) |
| `scripts/main.ts` | 핵심 실행 로직 | O (실행 시) |
| `scripts/helper.ts` | 유틸리티 함수 | O (필요 시) |
| `references/api-guide.md` | API 키, 엔드포인트, ID 매핑 테이블 | O (필요 시) |
| `.env` | 환경변수 (API 키 값 등) | X (Bun이 자동 로드) |

### references/ 폴더 활용

`references/` 폴더는 에이전트가 업무 수행에 필요한 **참고 자료**를 모아두는 곳입니다.

```markdown
# references/api-guide.md

## Notion API
- Base URL: https://api.notion.com/v1
- API Key: .env의 NOTION_API_KEY 사용
- 매출 대시보드 DB ID: `abc123def456`
- 일정 DB ID: `789ghi012jkl`

## Slack API
- Base URL: https://slack.com/api
- Bot Token: .env의 SLACK_BOT_TOKEN 사용
- 채널 ID 매핑:
  | 채널명 | ID |
  |--------|-----|
  | #morning-briefing | C01ABC123 |
  | #ops-alert | C02DEF456 |
  | #issues | C03GHI789 |

## Notion 블록 ID 매핑
- 커머스 매출 페이지: `page_abc123`
- 교육 매출 페이지: `page_def456`
- 컨설팅 매출 페이지: `page_ghi789`
```

이렇게 해두면 에이전트가 "Slack #morning-briefing에 보내야 하는데, 채널 ID가 뭐지?" 할 때 `references/api-guide.md`를 참조하여 `C01ABC123`을 사용합니다.

## 5. 스킬 로드 우선순위

에이전트가 스킬을 찾을 때의 우선순위입니다.

```
1. Workspace  →  <workspace>/skills/      (현재 프로젝트 전용)
2. Shared     →  ~/.openclaw/skills/      (모든 프로젝트 공용)
3. Bundle     →  내장 스킬                  (OpenClaw 기본 제공)
```

| 레벨 | 경로 | 용도 |
|------|------|------|
| **Workspace** | `<workspace>/skills/` | 이 프로젝트에서만 쓰는 스킬 |
| **Shared** | `~/.openclaw/skills/` | 여러 프로젝트에서 공통으로 쓰는 스킬 |
| **Bundle** | (내장) | OpenClaw이 기본 제공하는 스킬 |

**같은 이름의 스킬이 여러 레벨에 있으면?** Workspace가 우선합니다. 프로젝트별로 스킬을 커스터마이징할 수 있다는 뜻입니다.

```
예: daily-briefing 스킬이 Workspace와 Shared 모두에 존재
→ Workspace 버전이 사용됨 (프로젝트별 커스터마이징)
```

## 6. n8n vs Skills — 언제 무엇을 쓸까

n8n을 완전히 버리라는 뜻이 아닙니다. 각각 적합한 영역이 있습니다.

### 판단 기준

| 기준 | n8n | Skills |
|------|:---:|:------:|
| 단순 반복 (A → B → C) | O | X |
| 외부 웹훅 릴레이 | O | X |
| 빠른 프로토타이핑 | O | X |
| 맥락 의존 판단 | X | O |
| 복잡한 예외 처리 | X | O |
| 다중 시스템 연동 | X | O |
| 점점 복잡해지는 업무 | X | O |
| 버전 관리 필요 | X | O |

### 핵심 차이

```
n8n:    트리거 → 블록A → 블록B → 블록C → 완료
        (정해진 경로를 따른다)

Skills: 에이전트가 SKILL.md를 읽고,
        상황을 판단하고,
        적절한 스크립트를 실행하고,
        결과에 따라 다음 행동을 결정한다.
        (맥락적으로 판단한다)
```

### 실전 예시: "고객 문의 처리"

**n8n으로 구현하면:**
```
웹훅 → 키워드 분류(if/else) → 답변 템플릿 선택 → 발송
```
- "환불"이라는 키워드가 있으면 환불 템플릿, "배송"이면 배송 템플릿
- 키워드에 없는 문의가 오면? → 분류 실패, 기본 템플릿 발송 (고객 불만)

**Skills로 구현하면:**
```
에이전트가 문의 내용을 읽고 → 맥락을 파악하고 →
"이 고객은 환불을 원하는 게 아니라 교환을 원하는 것 같다" →
교환 절차 안내 + 필요한 정보 요청
```
- 에이전트가 SKILL.md의 예외 처리 가이드를 참고하여 맥락적으로 판단

## 7. 실전 스킬 예시 3개

### 예시 1: 리더 등록 스킬

```
~/.openclaw/skills/register-leader/
├── SKILL.md
├── scripts/
│   └── main.ts
└── references/
    └── api-guide.md
```

**SKILL.md:**
```markdown
---
name: register-leader
description: 새로운 리더를 Notion DB에 등록하고 Slack으로 안내 메시지 발송
---

# 리더 등록 스킬

## 개요
새로운 리더(강사/멘토) 정보를 Notion 데이터베이스에 등록하고,
해당 리더에게 Slack DM으로 온보딩 안내를 발송합니다.

## 필요한 정보
- 이름 (필수)
- 이메일 (필수)
- 전화번호 (필수)
- 전문 분야 (필수)
- Slack 사용자 ID (선택 — 없으면 이메일로 조회)

## 실행 순서

### 1단계: 입력 검증
- 필수 항목 누락 시 → 사용자에게 재요청
- 이메일 형식 검증
- 전화번호 형식 통일 (010-xxxx-xxxx)

### 2단계: 중복 확인
- Notion "리더" DB에서 이메일로 검색
- 이미 존재하면 → "이미 등록된 리더입니다. 정보를 업데이트할까요?" 확인

### 3단계: Notion 등록
- `scripts/main.ts`의 `registerLeader()` 함수 실행
- 등록 성공 시 → 생성된 페이지 URL 반환

### 4단계: Slack 안내 발송
- 리더의 Slack ID로 DM 발송
- 온보딩 안내 메시지 (가이드 문서 링크, 첫 미팅 일정 등)

## 예외 처리
- Notion API 실패 → 3회 재시도 후 사용자에게 수동 등록 요청
- Slack ID를 찾을 수 없음 → 이메일로 대체 안내 (생략 가능)
```

**scripts/main.ts:**
```typescript
const NOTION_API_KEY = process.env.NOTION_API_KEY!;
const LEADER_DB_ID = "abc123def456";

interface LeaderInfo {
  name: string;
  email: string;
  phone: string;
  expertise: string;
  slackId?: string;
}

export async function registerLeader(info: LeaderInfo) {
  // 중복 확인
  const existing = await searchByEmail(info.email);
  if (existing) {
    return { status: "duplicate", page: existing };
  }

  // Notion 페이지 생성
  const response = await fetch("https://api.notion.com/v1/pages", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${NOTION_API_KEY}`,
      "Content-Type": "application/json",
      "Notion-Version": "2022-06-28",
    },
    body: JSON.stringify({
      parent: { database_id: LEADER_DB_ID },
      properties: {
        "이름": { title: [{ text: { content: info.name } }] },
        "이메일": { email: info.email },
        "전화번호": { phone_number: info.phone },
        "전문 분야": { select: { name: info.expertise } },
      },
    }),
  });

  const page = await response.json();
  return { status: "created", page };
}

async function searchByEmail(email: string) {
  const response = await fetch(`https://api.notion.com/v1/databases/${LEADER_DB_ID}/query`, {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${NOTION_API_KEY}`,
      "Content-Type": "application/json",
      "Notion-Version": "2022-06-28",
    },
    body: JSON.stringify({
      filter: {
        property: "이메일",
        email: { equals: email },
      },
    }),
  });

  const data = await response.json();
  return data.results.length > 0 ? data.results[0] : null;
}
```

### 예시 2: 이벤트 관리 스킬

```
~/.openclaw/skills/manage-event/
├── SKILL.md
├── scripts/
│   ├── main.ts
│   └── notify.ts
└── references/
    └── api-guide.md
```

**SKILL.md:**
```markdown
---
name: manage-event
description: 교육/세미나 이벤트 생성, 참가자 관리, 리마인더 발송
---

# 이벤트 관리 스킬

## 개요
교육, 세미나, 워크숍 등 이벤트를 생성하고 참가자를 관리합니다.

## 지원하는 작업
1. **이벤트 생성**: Notion에 이벤트 페이지 생성 + Google Calendar 등록
2. **참가자 등록**: 이벤트에 참가자 추가 + 확인 메시지 발송
3. **리마인더 발송**: 이벤트 D-1, D-day 리마인더 Slack/이메일 발송
4. **참가자 목록 조회**: 특정 이벤트의 참가자 목록 반환

## 실행 순서 — 이벤트 생성

### 1단계: 정보 수집
- 이벤트명, 날짜, 시간, 장소, 정원, 설명
- 누락 항목 있으면 사용자에게 질문

### 2단계: Notion 등록
- "이벤트" DB에 새 페이지 생성
- 상태: "모집중"

### 3단계: Google Calendar 등록
- `scripts/main.ts`의 `createCalendarEvent()` 실행
- 참석자에게 캘린더 초대 발송

### 4단계: 안내 메시지 생성
- 이벤트 안내 메시지를 생성하여 사용자에게 전달
- 사용자가 확인 후 발송 채널 지정

## 예외 처리
- 동일 날짜/시간에 이벤트 중복 → 사용자에게 확인 요청
```

### 팁 2: 예외 처리를 반드시 적으세요

에이전트가 실행할 수도 있지만, 사람이 직접 `bun run scripts/main.ts`로 테스트할 수도 있어야 합니다.

## 심화: SKILL.md 전체 frontmatter

기본적인 `name`과 `description` 외에도, SKILL.md의 frontmatter에는 다양한 고급 옵션을 지정할 수 있습니다.

```yaml
---
name: my-skill
description: Brief explanation
homepage: https://example.com
user-invocable: true
disable-model-invocation: false
command-dispatch: tool
command-tool: exec
command-arg-mode: raw
metadata:
  openclaw:
    requires:
      bins: ["jq", "curl"]
      env: ["MY_API_KEY"]
    os: ["darwin", "linux"]
    emoji: "🔧"
    primaryEnv: MY_API_KEY
    install:
      brew: { formula: "my-tool" }
---
```

### 주요 필드 설명

| 필드 | 설명 |
|------|------|
| `user-invocable: true` | 사용자가 `/my-skill`처럼 직접 호출 가능. `false`면 다른 스킬이나 에이전트를 통해서만 실행 |
| `disable-model-invocation: true` | LLM이 자동으로 이 스킬을 호출하는 것을 차단. 사용자 명시적 호출만 허용 |
| `command-dispatch: tool` | 모델을 우회하여 직접 실행. LLM 토큰을 소비하지 않고 스크립트를 바로 실행 |
| `command-tool: exec` | 실행에 사용할 도구 지정 |
| `command-arg-mode: raw` | 인자를 가공 없이 그대로 전달 |

### requires: 의존성 자동 체크

```yaml
requires:
  bins: ["jq", "curl", "bun"]    # 시스템에 설치되어 있어야 하는 바이너리
  env: ["MY_API_KEY", "SLACK_TOKEN"]  # 설정되어 있어야 하는 환경변수
```

스킬 로드 시 `bins`에 나열된 바이너리와 `env`에 나열된 환경변수가 존재하는지 자동으로 확인합니다. 누락된 항목이 있으면 에이전트에게 알려주고, `install` 섹션에 설치 방법이 명시되어 있으면 자동 설치를 시도합니다.

### {baseDir} 변수

스크립트 경로를 지정할 때 `{baseDir}`을 사용하면 스킬 폴더의 절대 경로로 자동 치환됩니다.

```markdown
## 실행 순서
1. `{baseDir}/scripts/main.ts` 실행
2. 결과를 `{baseDir}/output/` 폴더에 저장
```

스킬이 Workspace(`<workspace>/skills/`)에 있든 Shared(`~/.openclaw/skills/`)에 있든 경로가 올바르게 해석됩니다.

## 심화: 스킬 토큰 비용

스킬은 무료가 아닙니다. 등록된 스킬마다 시스템 프롬프트에 오버헤드가 추가됩니다.

### 스킬당 오버헤드

```
스킬 1개 등록 시:
  이름 (name):        ~4 토큰
  설명 (description): ~20 토큰
  ──────────────────────────
  합계:              ~24 토큰/스킬
```

24토큰은 미미해 보이지만, 스킬이 50개면 1,200토큰, 100개면 2,400토큰입니다. 매 API 호출마다 이 비용이 발생합니다.

### 스킬이 많을수록 시스템 프롬프트 증가

```
스킬 10개:  ~240 토큰 추가
스킬 50개:  ~1,200 토큰 추가
스킬 100개: ~2,400 토큰 추가
```

Prompt Caching이 적용되면 캐시 HIT 시 비용이 90% 절감되지만, 캐시 MISS(스킬 목록이 변경될 때)에는 원래 비용이 발생합니다.

### 불필요한 스킬 비활성화

자주 사용하지 않는 스킬은 `openclaw.json`의 `entries`에서 `enabled: false`로 설정하세요.

```json5
skills: {
  entries: {
    "rarely-used-skill": { enabled: false },
    "seasonal-skill": { enabled: false }
  }
}
```

비활성화된 스킬은 시스템 프롬프트에 포함되지 않아 토큰을 절약합니다. 필요할 때 `enabled: true`로 다시 활성화하면 됩니다.

### 커뮤니티 규모

- **13,729+ 커뮤니티 스킬** 등록 (2026년 3월 기준)
- 카테고리: 마케팅, 개발, 데이터 분석, 고객 응대, 콘텐츠 제작 등

### 보안 검증 3단계

| 단계 | 검증 방법 | 설명 |
|------|----------|------|
| 1단계 | Auto Scan | VirusTotal + OpenClaw AI가 자동으로 코드 분석. 악성 패턴, 의심스러운 API 호출, 데이터 유출 시도 탐지 |

3단계를 모두 통과한 스킬에는 "Verified" 배지가 부여됩니다. 검증되지 않은 스킬도 설치할 수 있지만, 경고 메시지가 표시됩니다.

## 심화: openclaw.json 스킬 설정

`openclaw.json`에서 스킬의 로드, 활성화, 환경변수를 세밀하게 제어할 수 있습니다.

```json5
skills: {
  entries: {
    "daily-briefing": { enabled: true, env: { API_KEY: "xxx" } },
    "cs-responder": { enabled: true, apiKey: { source: "env", id: "CS_KEY" } }
  },
  allowBundled: ["weather", "image"],
  load: { watch: true, watchDebounceMs: 250, extraDirs: ["/extra/skills"] }
}
```

### entries: 개별 스킬 설정

| 필드 | 설명 |
|------|------|
| `enabled` | 스킬 활성화 여부. `false`면 시스템 프롬프트에서 제외 |
| `env` | 해당 스킬 전용 환경변수. 글로벌 `.env`보다 우선 |
| `apiKey` | API 키 소스 지정. `source: "env"`면 시스템 환경변수에서, `source: "file"`이면 파일에서 읽음 |

### allowBundled: 내장 스킬 선택적 허용

OpenClaw이 기본 제공하는 내장(Bundle) 스킬 중 사용할 것만 명시합니다. 빈 배열(`[]`)이면 내장 스킬을 모두 차단합니다.

### load: 스킬 로드 설정

| 필드 | 설명 |
|------|------|
| `watch: true` | 스킬 폴더의 파일 변경을 감지하여 자동 리로드 |
| `watchDebounceMs: 250` | 파일 변경 감지 후 250ms 대기 후 리로드 (빈번한 저장 시 중복 리로드 방지) |
| `extraDirs` | 기본 경로(`<workspace>/skills/`, `~/.openclaw/skills/`) 외에 추가로 스킬을 로드할 폴더 |

`extraDirs`는 팀 공유 스킬 폴더를 지정할 때 유용합니다. 예를 들어 NAS나 공유 드라이브에 있는 스킬 폴더를 가리킬 수 있습니다.

> 다음 장에서는 여러 스킬을 조합하여 복잡한 업무를 자동화하는 **워크플로우 오케스트레이션**을 다룹니다.
